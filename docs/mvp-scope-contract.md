# チャイ研究アプリ MVP Scope Contract

**作成日:** 2026-04-19

この文書は、チャイ研究アプリv1 MVPの実装範囲を固定する契約文書である。企画書、要件定義書、詳細設計書、UI設計書、技術スタック、プロジェクト方針の間で解釈が揺れた場合、v1のスコープ判断は本書を優先する。

本書は新しい機能を追加するための文書ではない。v1を、非公開の個人研究ログとして安全に完成させるため、やること・やらないこと・判断手順を明確にする。

## 1. v1 MVPの核

v1 MVPの目的は、ユーザーが自分のチャイ作りを試行単位で記録し、前回の試行を複製して少し変え、履歴から読み返せる状態を作ることである。

v1では、以下の原則を固定する。

1. 完成レシピではなく、1回ごとの試行を主役にする。
2. すべての業務データを非公開にする。
3. 公開、共有、SNS、AI提案は扱わない。
4. 写真、カスタム項目、複雑な比較、系譜グラフは扱わない。
5. オフライン自動同期は扱わず、軽量なローカル下書きに限定する。
6. DB、RLS、UI、受け入れ基準がそろっていない機能は実装しない。

## 2. v1で実装するもの

v1 MVPに含める機能は以下に限定する。

| ID | 機能 | v1での成立条件 |
|---|---|---|
| MVP-01 | Magic Link認証 | Supabase Authでログインでき、未認証ユーザーは業務データを閲覧できない。 |
| MVP-02 | 研究ライン作成・編集・アーカイブ | 研究テーマを作成、編集、アーカイブできる。物理削除はv1では実装しない。アーカイブ済み研究ラインは新規試行選択から外し、既存試行は履歴・詳細から参照できる。 |
| MVP-03 | 試行作成・編集・論理削除 | 研究ラインに紐づく試行を保存し、編集し、`deleted_at` による論理削除ができる。 |
| MVP-04 | 材料行入力 | 茶葉、水、ミルク、甘味、スパイス、その他を材料行として保存できる。 |
| MVP-05 | 試行複製 | 自分の過去試行を複製し、新しい試行の `parent_trial_id` に元試行IDを保存できる。 |
| MVP-06 | 試行履歴 | 自分の試行を最新順で一覧し、研究ライン、スター、日付範囲で絞り込める。 |
| MVP-07 | 親試行リンク | 試行詳細で元にした試行をリンク表示できる。グラフ表示はしない。 |
| MVP-08 | スター | 自分の試行にスターを付け外しでき、履歴で絞り込める。 |
| MVP-09 | 軽量下書き | 入力中の試行を同一ブラウザ内に一時保存し、ユーザー操作で復元または破棄できる。 |
| MVP-10 | RLSによるデータ隔離 | 他ユーザーの研究ライン、試行、材料行、スターを参照・更新できない。 |
| MVP-11 | モバイルファーストUI | 主要画面をモバイル1カラムで操作でき、保存失敗時も入力内容を保持する。 |

## 3. v1で実装しないもの

以下はv1 MVPから除外する。ユーザーから依頼された場合でも、設計文書の更新と人間レビューなしに実装してはならない。

| 対象外機能 | v1で却下する理由 | 代替 |
|---|---|---|
| 公開・限定公開・共有URL | RLS、URL設計、権限境界、削除時挙動が複雑になる。 | すべて非公開として保存する。 |
| SNS要素 | プロダクトの核が研究ログから投稿・交流へぶれる。 | v1では個人利用に限定する。 |
| 写真アップロード | Storage/R2、容量、削除、プライバシー対応が必要になる。 | テキストと材料行だけで記録する。 |
| カレンダービュー | 初期価値に対してUI・集計コストが高い。 | 最新順の履歴と日付絞り込みで代替する。 |
| 任意2件比較・複数件比較・グラフ | 比較定義が複雑で、材料行の正規化も不足する。 | 親試行リンクで前回との差を人が確認する。 |
| 本格的な系譜グラフ | 可視化コストが高く、実用性の検証が先に必要である。 | `parent_trial_id` の保存とリンク表示に留める。 |
| お気に入り棚・定番昇格 | スターとの概念差が曖昧になりやすい。 | v1はスターのみ実装する。 |
| カスタム項目 | 検索、比較、表示、型管理を壊しやすい。 | 基本項目と最小限の詳細項目だけ扱う。 |
| 材料プリセット・スパイスブレンド | 材料マスター設計が必要になりMVPが重くなる。 | 試行ごとの材料行をスナップショット保存する。 |
| オフライン自動同期 | 競合解決、再送、複数端末同期が必要になる。 | 同一ブラウザ内の軽量下書きだけ扱う。 |
| AI提案 | 外部送信、プロンプト設計、誤提案、コスト管理が必要になる。 | v1ではユーザー自身の記録に集中する。 |

## 4. v1のデータ境界

v1で作成してよい業務テーブルは以下に限定する。

| テーブル | 用途 |
|---|---|
| `research_lines` | 研究テーマを保存する。 |
| `trials` | 1回分の試行ログを保存する。 |
| `trial_ingredients` | 試行時点の材料行を保存する。 |
| `trial_stars` | 試行へのスターを保存する。 |

以下のテーブルまたは同等の構造はv1では作成しない。

- 公開設定、共有URL、フォロー、コメント、リアクションに関するテーブル
- 写真、ファイル、画像メタデータに関するテーブル
- 材料マスター、スパイスブレンド、プリセットに関するテーブル
- カスタム項目定義、フォームテンプレート、評価テンプレートに関するテーブル
- 比較結果、統計、系譜グラフ専用に関するテーブル

## 5. v1のUI境界

v1で作成してよい主要画面は以下に限定する。

| 画面ID | 画面 |
|---|---|
| A1 | 認証画面 |
| H1 | ホーム |
| L1 | 研究ライン一覧 |
| L2 | 研究ライン詳細 |
| T1 | 試行入力フォーム |
| T2 | 試行詳細 |
| T3 | 試行履歴 |
| S1 | 最小設定 |

以下の画面または導線はv1では作成しない。

- 比較画面
- 系譜ビュー
- カレンダービュー
- 写真管理画面
- 公開設定画面
- 共有URL管理画面
- AI提案画面
- SNS的なプロフィール、フォロー、コメント画面

## 6. スコープ判断ルール

新しい要望が出た場合は、実装前に次の順で判定する。

1. その要望は、試行を記録、複製、読み返す体験に直接必要か。
2. v1の非公開・単独利用の前提を崩さないか。
3. 既存4テーブルとRLSで安全に表現できるか。
4. 既存画面IDの中で自然に扱えるか。
5. 画面別受け入れ基準を満たす形で検証できるか。
6. 新しいライブラリ、Storage、Realtime、API Routes、外部AIを必要としないか。

1つでも満たせない場合は、v1実装を却下し、代替案またはv2以降の設計課題として扱う。

## 7. スコープ変更手順

v1の範囲を変更する場合は、実装前に以下をすべて更新する。

1. 本書の実装対象または対象外機能。
2. 要件定義書の機能要件。
3. 詳細設計書のDB、RLS、データアクセス方針。
4. UI設計書または画面別受け入れ基準。
5. DB Migration & RLS Policy。
6. Supabase Data Access & Error Contract。
7. Deployment Contract。
8. Codex Execution Rules。

スコープ変更は、文書更新、影響範囲の説明、人間レビューを終えるまで着手してはならない。

## 8. v1完了条件

v1 MVPは、以下を満たしたときに完了とする。

1. 認証、研究ライン、試行、材料行、複製、履歴、スター、下書きが動作する。
2. 他ユーザーのデータを読み書きできないことをRLSテストで確認している。
3. 主要画面がモバイルとデスクトップで破綻していない。
4. 保存失敗時に入力内容を保持し、ユーザーが再試行できる。
5. 公開、写真、比較、系譜、カスタム項目、AI、オフライン自動同期のUIとDBが存在しない。
6. Supabase操作とエラー表示が契約文書に沿っている。
7. Codexまたは開発者が、範囲外機能を勝手に追加していない。
