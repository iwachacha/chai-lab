# チャイ研究アプリ v1 実装計画書 監査報告

**作成日:** 2026-04-19  
**監査対象:** `docs/implementation-plan-v1.md`  
**監査者の立場:** 実装担当ではなく、仕様逸脱・安全性・検証不足を検出する監査者

## 1. 監査の目的

この監査の目的は、`docs/implementation-plan-v1.md` が、チャイ研究アプリv1を安全に実装開始できる計画書として十分かを検証することである。特に、非公開の個人研究ログというv1価値、RLSによるユーザー隔離、RPCへの書き込み集約、Next.js静的構成、スコープ厳守、テスト可能性を壊す余地がないかを確認した。

監査対象は、以下の1ファイルである。

- `docs/implementation-plan-v1.md`

監査基準は以下の文書である。

- `docs/mvp-scope-contract.md`
- `docs/pj-policy.md`
- `docs/app-rdd.md`
- `docs/app-lld.md`
- `docs/app-design.md`
- `docs/screen-acceptance-criteria.md`
- `docs/tech-stack.md`
- `docs/db-migration-rls-policy.md`
- `docs/supabase-data-access-error-contract.md`
- `docs/deployment-contract.md`
- `docs/codex-execution-rules.md`
- `README.md`

監査時の前提として、現在の計画書は実装コードではなく、実装開始前の作業計画である。そのため、SQLやコンポーネントの未実装自体は欠陥ではない。ただし、計画書の粒度や順序が原因で、実装者がRLS検証前にUI実装へ進める、DB変更を粗いmigrationにまとめる、直接CRUDを書く、対象外機能を混ぜる、受け入れ基準を満たさず完了扱いにする余地がある場合は欠陥として扱う。

## 2. 総合評価

- 総合評価: 主要修正完了まで実装開始不可
- 一言要約: スコープ意識は強いが、DB/RLS/RPCのゲート、migration粒度、人間レビュー、受け入れ基準トレース、エラー正規化の具体性が不足しており、このまま実装を始めるとRLS事故または大きな手戻りを起こす。
- 重大欠陥件数: 3件
- 主要欠陥件数: 16件
- 軽微欠陥件数: 6件
- 今のまま実装を始めてよいかの判断: 不可。少なくともCritical 3件と、DB/RLS/RPC・エラー・デプロイに関するMajorを先に修正する必要がある。

## 3. 総合所見

この計画書の強い点は、v1の対象外機能を広く明記し、公開、写真、AI、比較、系譜、カスタム項目、オフライン自動同期を排除しようとしている点である。M0〜M8の大枠も、認証、DB/RLS、RPC、UI、テスト、デプロイの順で進める意図は読み取れる。

危険な点は、DB/RLS/RPCの実装順と検証ゲートが計画書内で十分に閉じていないことである。M2-01でv1の4業務テーブル、RLS helper、権限まで作る想定に見える一方、M2直後に検証されるのは研究ラインRLSだけであり、`trials`、`trial_ingredients`、`trial_stars` のRLS・直接書き込み禁止・A/B分離はM7まで後ろに下がっている。これは `db-migration-rls-policy.md` の「migration、RLS、テスト観点をセットで扱う」原則に対して弱い。

最も実装事故につながりやすい箇所は、M2-01、M3-01、M4-01、M4-02である。これらはDB/RLS/security definer/RPCに触れるにもかかわらず、タスク完了条件に人間レビュー、`search_path` 固定、PUBLIC権限剥奪、GRANT範囲、SQLSTATE/エラー正規化、A/B分離の実行完了が明示されていない。

スコープ逸脱を招きやすい箇所は、直接的な機能追加よりも「基盤」「将来の運用」「テスト補助」の名目で起きる。具体的には、バックアップ/エクスポート、ロギング、E2E認証ヘルパー、UI共通部品、Radix追加、日時処理、下書き保存の周辺で、契約文書の禁止事項に触れない範囲が曖昧である。

先に直すべき箇所は、RLS/RPCゲート、migration分割、人間レビュー条件、AppResult/AppErrorの厳密化、受け入れ基準との対応表、静的export設定のM1完了条件化である。

## 4. 章別監査結果

### 1. 文書の目的

- 評価: B
- 判定理由: 計画書の目的、v1前提、扱う範囲は明確であり、コードを書かない前提も示されている。
- 問題点一覧: 監査上の重大問題は少ない。ただし「安全に実装開始」の定義が、この章ではDB/RLS/RPC検証ゲートまで含むとは明文化されていない。
- 欠けている内容: 「この計画書に従っても、Criticalな未確定事項がある場合は実装開始不可」という宣言。
- 曖昧な表現: 「実装開始前の計画」という表現は正しいが、実装開始可否の判断基準まではここで示していない。
- 実装者が危険に誤解しうる箇所: 章単体では少ない。
- 修正方針: 目的章に「本計画書は未確定事項の解消を前提とし、Criticalな未確定事項が残る場合は実装開始不可」と追記する。

### 2. 参照文書と優先順位

- 評価: B
- 判定理由: 必須文書は列挙され、MVP Scope Contract優先も反映されている。
- 問題点一覧: READMEとの「矛盾」を後続章で扱っているが、READMEは `mvp-scope-contract.md` をスコープ契約として説明しており、実質的な矛盾とは言いにくい。優先順位に関する記述はおおむね正しいが、DB変更レビューや人間レビューの優先度が計画全体へ十分に落ちていない。
- 欠けている内容: DB/RLS/security definer変更は `db-migration-rls-policy.md` のレビュー必須条件を優先するという明示。
- 曖昧な表現: 「専門領域の詳細が衝突する場合」という表現は正しいが、レビュー必須条件まで含むかが曖昧。
- 実装者が危険に誤解しうる箇所: DB/RLSの文書を参照しているだけで、人間レビューゲートを満たしたと誤解する可能性がある。
- 修正方針: DB/RLS/RPCに関する変更は、スコープ内でもレビュー必須条件を満たすまで完了扱いにしないと明記する。

### 3. v1の実装対象と非対象

- 評価: B
- 判定理由: v1対象/非対象の切り分けは広く記載され、範囲外機能の列挙も参照文書に沿っている。
- 問題点一覧: データエクスポートを「後でやる」に逃がす記述は妥当だが、`app-rdd.md` が要求する開発・運用側のSupabaseバックアップ状況と手動エクスポート手順の確認が計画に落ちていない。ユーザー向け機能としては対象外でも、運用確認としては必要である。
- 欠けている内容: 外部Analyticsを導入しないこと、ログに材料名・メモ・Magic Link・トークンを含めないこと。
- 曖昧な表現: 「本格的なバックアップ、復元」はv2扱いとしているが、運用側のバックアップ確認まで後回しにしてよいように読める。
- 実装者が危険に誤解しうる箇所: 「データエクスポートはv1では保留」を根拠に、Supabaseバックアップ確認や手動エクスポート手順確認まで省略する可能性がある。
- 修正方針: ユーザー向けエクスポート機能は非対象、運用側バックアップ/手動エクスポート確認はM0またはM8対象、と明確に分ける。

### 4. 実装開始前の確認事項

- 評価: C
- 判定理由: 未確定事項は複数拾えているが、止める条件が不足している。特に研究ライン名正規化、clone_trial UX、バックアップ/ログ、DBレビュー必須条件が止める条件に入っていない。
- 問題点一覧: READMEとの優先順位差分を「矛盾」に近い扱いにしているが、これは実質的な問題ではない。一方で、`clone_trial` が即DB作成することによる「複製して編集」導線の放置試行リスク、`brewed_at` のtimestamptzとUIの日付入力・日付範囲の扱い、RPCエラー分類が保存以外の読み取りにも影響する点が弱い。
- 欠けている内容: DB/RLS/security definer変更の人間レビュー未完了時は止める、日時方針未決定なら履歴・日付絞り込みを止める、ログ方針未決定なら外部ロギングを入れない、という条件。
- 曖昧な表現: 「実装時に具体化が必要」は弱い。エラー正規化は実装時に何となく決めるものではなく、DB/RPC実装前に分類方針を決める必要がある。
- 実装者が危険に誤解しうる箇所: RPCエラー分類はUI保存処理だけ止めればよいと読めるが、詳細取得、一覧取得、clone、soft_delete、starにも影響する。
- 修正方針: 未確定事項ごとに「止める作業」を紐づけ、研究ライン重複、日時、RPCエラー分類、人間レビュー、運用バックアップ確認を明示的な停止条件にする。

### 5. 全体実装戦略

- 評価: B
- 判定理由: DB/RLS/RPCとデータアクセスを先に固める方針は正しい。
- 問題点一覧: 方針としては正しいが、後続のM2/M7に落としたとき、全テーブルRLSテストがDB作成直後に実行されない構造になっている。戦略とタスクの整合が崩れている。
- 欠けている内容: DB/RLS/RPC変更は実装PR単位でレビューとテストを完了させる、というゲート。
- 曖昧な表現: 「RPCの契約とテストを先に作ってからUIに接続する」はよいが、どのテストを「先に」とみなすかが曖昧。
- 実装者が危険に誤解しうる箇所: M7でまとめてテストすれば、M3/M4では暫定的に進めてよいと誤解しうる。
- 修正方針: 「M2/M3/M4のDB/RLS/RPCタスクは、それぞれのRLS/RPCテスト完了を次タスクの着手条件にする」と追記する。

### 6. マイルストーン計画

- 評価: C
- 判定理由: 大枠の順序は妥当だが、RLS/RPC検証がM7に寄りすぎており、DB作成からUI実装までの安全ゲートが不足している。
- 問題点一覧: M2は研究ラインのマイルストーンだが、M2-01ではv1コア4テーブルを作る。マイルストーン名と実際のDB影響範囲が一致していない。M3/M4でRPCを作る前に、`trials`、`trial_ingredients`、`trial_stars` の基本RLS・権限が検証済みであることが必要だが、M2の完了条件に入っていない。
- 欠けている内容: M2完了条件に「4テーブル全体のRLSと直接書き込み禁止テストが通っていること」を入れるべき。
- 曖昧な表現: M2の依存関係「DB/RLS基盤」は、どの範囲が完了済みか不明。
- 実装者が危険に誤解しうる箇所: 研究ラインだけ動けばM2完了としてM3へ進み、試行系RLS未検証のままRPC/UIへ進む可能性がある。
- 修正方針: M2を「DB/RLS基盤 + 研究ライン」に再定義するか、M2aとして4テーブルRLS/権限検証を独立させる。

### 7. 詳細タスク分解

- 評価: C
- 判定理由: タスク数と形式は十分だが、DB/RLS/RPC周辺の粒度とゲートが危険である。UIタスクにも受け入れ基準の必須項目が一部落ちている。
- 問題点一覧: M2-01が大きすぎる。M2-02は研究ラインRLSのみで、試行・材料・スターRLSの直後検証がない。M3-01/M4-01/M4-02はsecurity definerの権限固定や人間レビューをタスク完了条件にしていない。M2-04はL1受け入れ基準の試行数・最終試行日が明示されていない。
- 欠けている内容: DB taskごとのレビュー必須条件、SQL権限確認、RLS全テーブルテスト、受け入れ基準チェックリスト。
- 曖昧な表現: 「migration単位で作る」「RLS helper、基本RLS、権限を作る」は作業単位として広すぎる。
- 実装者が危険に誤解しうる箇所: 1つの巨大migrationで4テーブル、helper、policy、grantまでまとめてもよいと読める。
- 修正方針: M2-01をテーブルDDL、インデックス、RLS helper、RLS policy、GRANT/REVOKE、全テーブルRLSテストに分割する。M3/M4のRPCタスクにはsecurity definer hardeningとレビュー完了を完了条件に入れる。

### 8. DB / RLS / RPC 実装方針

- 評価: C
- 判定理由: 方針は参照文書に沿っているが、実装事故を防ぐ具体策が不足している。
- 問題点一覧: 直接CRUD禁止の検索例が `insert` と `update` 中心で、`delete`、`upsert`、`from('trial_ingredients').delete`、任意のテーブル名ラッパー、SQL関数経由の迂回を検出する計画が弱い。RPCの扱いに `search_path`、PUBLIC revoke、authenticated grantは章8.3にあるが、RPC各タスクには落ちていない。
- 欠けている内容: DB変更ごとの人間レビュー、rollback方針、GRANT/REVOKEテスト、security definerのsearch_path固定確認。
- 曖昧な表現: 「コードレビューと検索」は検索パターンと実行タイミングが不足している。
- 実装者が危険に誤解しうる箇所: データアクセス層なら直接CRUDを書いてもレビューで見逃される可能性がある。
- 修正方針: direct CRUD禁止の検査対象を明示し、CIまたはレビュー手順に検索コマンドを入れる。RPCごとの権限テストを追加する。

### 9. データアクセスとエラーハンドリング方針

- 評価: C
- 判定理由: UIからSupabaseを直接呼ばない方針はあるが、`AppResult` / `AppError` の厳密性が不足している。
- 問題点一覧: `AppResult相当` と書かれており、契約文書の `retryable`、`fieldErrors`、ユーザー向け `message`、Supabase生エラー非表示が型として固定されていない。RPC SQLSTATEの未確定を認識しているが、保存以外の読み取り・スター・clone・deleteにも波及することが明示されていない。
- 欠けている内容: エラー分類マッピング表、fieldErrorsの扱い、retryableの決定規則、ユーザー向け文言と開発者ログの分離。
- 曖昧な表現: 「相当」「寄せる」は実装者によって解釈が割れる。
- 実装者が危険に誤解しうる箇所: 生のSupabaseエラーをmessageに入れても `AppError相当` と見なす実装が可能になる。
- 修正方針: `AppResult<T>` と `AppError` の完全な型、全データアクセス関数のエラー分類、RPC別のマッピング表を計画に追加する。

### 10. UI実装方針

- 評価: C
- 判定理由: モバイル1カラムや禁止導線は書かれているが、実装順がDB/RPC依存を誤読させる。
- 問題点一覧: 画面の実装順だけを見ると、T1、T2、T3をUI都合で進められるように読める。M3/M4のRPC完了前にT1/T2を完成扱いにしない条件が章10にはない。デザイントークンをTailwindへどう反映するか、任意色をどう検出するかも不足している。
- 欠けている内容: 画面ごとの受け入れ基準対応表、デザイントークン実装タスク、任意色/影追加の監査手順。
- 曖昧な表現: 「最大幅を保ち」「影は控えめ」は、実装者の主観で広がる。
- 実装者が危険に誤解しうる箇所: 共通UI整備の名目で余計なカード階層、装飾、未来導線を入れる可能性がある。
- 修正方針: UI章に「UI画面は対応するData Access/RPC完了後に接続し、未接続の仮UIを完了扱いしない」と追記し、画面別受け入れ基準チェック表を追加する。

### 11. テスト戦略

- 評価: C
- 判定理由: テスト種別は網羅的だが、実行タイミングとゲートが弱い。
- 問題点一覧: RLS/RPC統合テストがM7に大きく置かれているため、DB/RPC変更直後の安全確認が後回しに読める。`主要テストが実行できる` は完了条件として曖昧であり、どのテストが必須で、どれが未実施許容なのかが不明。
- 欠けている内容: M2/M3/M4ごとの必須テスト、CIまたは手動実行コマンド、失敗時の停止条件。
- 曖昧な表現: 「主要テスト」「最低限確認」「代替確認」は、計画書単体では判定できない。
- 実装者が危険に誤解しうる箇所: RLSテストを最後にまとめて走らせるだけで十分と誤解する。
- 修正方針: RLS/RPCテストをM7だけでなく、DB/RPCタスク直後の完了条件に移す。テスト実行リストと未実施時のリリース不可条件を明記する。

### 12. リスク一覧

- 評価: B

- 問題点一覧: DB人間レビュー漏れ、Supabaseバックアップ/手動エクスポート未確認、ログへの研究内容混入、日時境界、AppErrorの曖昧実装、巨大migrationのrollback困難がリスクとして載っていない。
- 欠けている内容: レビュー未完了、ログ漏えい、日時ズレ、バックアップ未確認、migration肥大化。
- 曖昧な表現: 「PRで監査する」は誰が何を確認するか不明。
- 実装者が危険に誤解しうる箇所: リスク表にないため、バックアップやログをv1計画外として扱う可能性がある。

### 14. 推奨実装順の要約

## 6. 横断的な重大論点

### v1範囲外が入り込む余地

直接的な公開、写真、AI、比較、系譜、カスタム項目は計画書で明確に禁止されている。一方で、ロギング、バックアップ、E2E補助、共通UI、Radix追加、下書き、日付処理の周辺で「基盤」名目の拡張が入り込む余地がある。特に外部Analyticsやログ送信については、計画書に明示的な禁止がない。

### RLS事故の余地

最大の問題である。M2-01で4テーブルを作る計画なのに、M2直後のRLSテストが研究ライン中心である。`trials`、`trial_ingredients`、`trial_stars` のselect、直接insert/update/delete拒否、他ユーザー操作拒否を、M3/M4より前に完了条件化する必要がある。

### 直接CRUDの余地

計画書は直接CRUD禁止を掲げているが、検出方法が狭い。`from('trials').insert` だけでなく、`update`、`delete`、`upsert`、`trial_ingredients` 全操作、抽象化関数経由、テスト用コードの混入まで確認しないと抜ける。

### RPC責務の曖昧さ

RPCの業務責務はおおむね書かれているが、security definerとしての安全要件、権限、search_path、PUBLIC revoke、エラー分類、SQLSTATEマッピング、レビュー必須条件がタスク単位に落ちていない。

### テスト不足の余地

テスト項目は多いが、実行タイミングが弱い。M7でまとめるのではなく、M2/M3/M4の完了条件にRLS/RPCテストを入れるべきである。CI・手動・未実施の区別も不足している。

### 静的構成との衝突

Deployment Contractは意識されているが、`next.config` の静的export、`trailingSlash`、固定ルート、クエリパラメータ方式はM1で固定すべきである。M8で発見すると手戻りが大きい。

### 無断ライブラリ追加の余地

禁止方針はあるが、Radix UI primitives追加の個別審査や、依存差分レビューの具体手順が弱い。特にDialog、Dropdown、Popover、Toastをまとめて入れる余地がある。

### 将来機能を呼び込みやすい構造

データエクスポート、バックアップ、ログ、下書き、日付処理は将来機能に近い。これらはv1外機能として切るものと、運用・安全上確認するものを分けなければ、実装者が過剰設計するか、逆に必要確認を省略する。

### 受け入れ基準との紐付け不足

画面別受け入れ基準を参照しているが、タスクごとの対応表がない。結果として、L1の試行数/最終試行日、アーカイブ確認文言、T1の単位候補、T3の50件単位、S1のログアウト時下書き確認などが個別に漏れうる。

### 実装順序による手戻りリスク

M1で静的export設定を厳密に固定しない、M2で全テーブルRLSを検証しない、M3/M4でRPC hardeningをタスク完了条件にしない、M7までRLS/RPC統合を待つ、という順序は手戻りが大きい。

## 7. 修正優先順位

### まず直すべきCritical

1. M2内で4業務テーブルすべてのRLS/権限/直接書き込み禁止テストを完了条件にする。
2. DB/RLS/security definer/RPC変更に人間レビュー完了ゲートを追加する。
3. M2-01をDDL、index、helper、policy、grant、RLSテストに分割し、巨大migrationを避ける。

### 次に直すべきMajor

1. RPCタスクへ `search_path` 固定、PUBLIC revoke、authenticated grant、所有者確認テストを明記する。

4. UIタスクと画面別受け入れ基準の対応表を追加する。

6. バックアップ/手動エクスポート確認、ログ方針、日時方針を実装前確認事項に追加する。

## 9. 実装開始可否判定

結論: 主要修正完了まで実装開始不可。

理由は、計画書の方向性はv1契約に沿っているものの、DB/RLS/RPCに関する安全ゲートが不足しているためである。特に、4業務テーブルを作るM2-01の直後に全テーブルRLS/権限/直接書き込み禁止テストが走らない構造は、非公開研究ログとして致命的な事故につながる。さらに、DB/RLS/security definer変更の人間レビュー必須条件がタスク完了条件にない点、巨大migrationに読めるM2-01の粒度、`AppResult` の曖昧さ、静的export設定の後回しは、実装前に修正しなければならない。

実装開始を許可できる最低条件は、Critical 3件の修正と、MajorのうちRPC hardening、AppError厳密化、直接CRUD禁止検査、静的export M1ゲート、受け入れ基準対応表の修正が完了していることである。
