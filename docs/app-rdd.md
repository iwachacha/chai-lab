# チャイ研究アプリ要件定義書 (v1)

この文書は、チャイ研究アプリのv1要件を定義する。v1は、非公開の個人研究ログとして、研究ライン、試行、複製、履歴、スターを安全に扱うことに集中する。公開、SNS、AI、写真、カスタム項目、オフライン自動同期はv1の対象外とする。

## 1. 目的と背景

### 1.1 目的

- チャイ作りの試行錯誤を記録し、前回から何を変えたかを振り返りながら、自分に合うチャイを継続的に研究できる環境を提供する。
- v1は個人の非公開研究ログに特化し、SNS的な公開・共有機能は扱わない。
- ユーザーは技術的知識を持たない前提とし、アプリは迷わず記録を続けられる入力体験を提供する。

### 1.2 背景

既存のレシピアプリや飲み物ログは、完成レシピの共有や単発の記録を主軸にしている。一方で、チャイ愛好家が継続的に知りたいのは、茶葉、水、ミルク、甘味、スパイス、煮出し時間などの違いが味や満足度にどう影響したかである。

本アプリは、完成レシピを公開する場ではなく、個人が試行を積み上げる研究ノートとして設計する。

## 2. 用語定義

| 用語 | 定義 |
|---|---|
| **研究ライン (research line)** | ユーザーが継続して研究するテーマや方向性。「朝用」「濃厚系」「生姜強め」など。v1では旧来の「レシピベース」という表現は使わず、研究ラインに統一する。 |
| **試行 (trial)** | 実際に作ったチャイ1回分の記録。材料、工程の一部、評価、メモ、次回の狙いを含む。 |
| **親試行 (parent trial)** | ある試行を複製または参考にして次の試行を作った場合の元試行。v1では系譜は親試行IDのみで管理する。 |
| **材料行 (trial ingredient)** | 試行に含まれる茶葉、水、ミルク、甘味、スパイスなどの記録行。カテゴリ、名称、量、単位を持つ。 |
| **スター (star)** | 後で見返したい試行に付ける軽い印。v1ではお気に入り棚や定番昇格とは区別し、スターのみを実装する。 |
| **下書き (draft)** | 入力途中の試行をブラウザ内に一時保存したもの。v1ではサーバー同期や複数端末同期は行わない。 |

## 3. システム概要

### 3.1 ユーザーロール

- **研究者 (標準ユーザー)**: チャイを作り、研究ラインと試行を記録し、過去試行を複製し、スターを付ける。v1では単独利用のみを対象とする。
- **管理者**: v1では専用の管理UIを持たない。DB運用や障害対応は開発・運用作業として扱う。

### 3.2 システム構成概要

1. **フロントエンド:** Next.js 静的アプリ。
2. **ホスティング:** Cloudflare Pages。
3. **認証:** Supabase Auth の Magic Link。
4. **データストア:** Supabase PostgreSQL。
5. **権限管理:** Supabase RLS によるユーザー単位のデータ隔離。
6. **サーバー側処理:** 試行保存、試行複製、試行論理削除など、整合性が必要な処理のみPostgres RPCで実装する。

v1では、Supabase Storage、Cloudflare R2、Realtime、GraphQL、外部AI API、オフライン自動同期は使用しない。

## 4. 機能要件

### 4.1 v1 MVPの基本機能

#### FR-001 ユーザー登録・ログイン

- Magic Linkを用いたメール認証を提供する。
- SNSログインはv2以降の検討対象とする。
- 認証済みユーザーは、自分の研究ラインと試行のみ閲覧・編集できる。

#### FR-002 研究ラインの作成

- ユーザーは研究ラインを作成・編集・アーカイブできる。
- 研究ラインは名前と任意の説明を持つ。
- v1では研究ラインに公開状態を持たせない。すべて非公開とする。
- v1では研究ラインの物理削除をUIから提供しない。
- アーカイブ済み研究ラインは新規試行選択から除外し、既存試行は履歴・詳細から参照できる。

#### FR-003 試行の登録

- ユーザーは研究ラインを選択して試行を登録できる。
- 試行は、試行名、日付、材料行、総合評価、一言メモ、次回の狙いを持つ。
- 材料行はカテゴリ、名称、量、単位を持つ。
- 詳細項目は、煮出し時間、沸騰回数、こし方など最小限に限定する。
- 試行本体と材料行の作成・編集は、部分保存を防ぐため `save_trial_with_ingredients` RPCに集約する。
- v1では写真、カスタム項目、評価テンプレートは扱わない。

#### FR-004 試行の複製

- ユーザーは自分の過去試行を複製して新しい試行を作成できる。
- 複製された試行は、元試行を `parent_trial_id` として保持する。
- 複製時は材料行、評価、メモ、詳細項目を初期値としてコピーし、ユーザーが変更できる。
- 作成日時、更新日時、スター状態はコピーしない。
- 複製は `clone_trial` RPCに集約し、他ユーザーの試行、論理削除済み試行、アーカイブ済み研究ラインの試行は複製できない。

#### FR-005 試行履歴

- ユーザーは自分の試行を最新順で一覧できる。
- 研究ライン、日付範囲、スター有無で絞り込みできる。
- v1ではカレンダービュー、グラフ表示、複数件分析は扱わない。

#### FR-006 スター

- ユーザーは自分の試行にスターを付けたり外したりできる。
- スターは軽い印であり、定番昇格やお気に入り棚とは扱いを分ける。
- v1ではスター付き試行の絞り込みのみ提供する。

#### FR-007 下書き保存

- 試行入力中の内容は、ブラウザ内に軽量な下書きとして保存できる。
- 下書きは同一端末・同一ブラウザでのみ復元対象とする。
- 下書きは認証ユーザー単位で分離し、未認証状態や別ユーザーでは復元しない。
- 下書きはサーバー保存やバックアップではないことをUIで明確にする。
- v1では下書きのサーバー同期、複数端末同期、自動再送は行わない。

### 4.2 v1では扱わない機能

以下はv2以降の検討対象とし、v1では実装しない。

- 公開・限定公開・共有URL
- SNS要素
- 写真アップロード
- カレンダービュー
- 任意2件比較、複数件比較、条件比較、グラフ表示
- 本格的な系譜ビュー
- カスタム項目
- 材料プリセット、スパイスブレンド
- お気に入り棚、定番昇格
- オフライン自動同期
- AI提案

## 5. 非機能要件

### 5.1 性能

- 主要画面の初期表示は通常回線で2秒以内を目標とする。
- v1では同一ユーザーあたり試行1,000件程度まで快適に扱えることを目標とする。
- 大量データ分析、グラフ表示、複数件比較はv1の性能要件に含めない。

### 5.2 セキュリティ

- v1の全データは非公開とする。
- すべての業務テーブルにRLSを適用し、ユーザー自身のデータのみ参照・更新できるようにする。
- 子テーブルは親レコード経由で所有者を確認する。
- 公開・限定公開に関するRLSはv1では実装しない。

### 5.3 保守性

- MVP範囲外の機能を先回りして実装しない。
- DB変更はmigrationとして管理し、RLSとテスト観点を合わせて記録する。
- カスタム項目や公開機能など、将来拡張の余地は残すが、v1のスキーマに不要な複雑性を持ち込まない。

### 5.4 ユーザビリティ

- 初回ユーザーが、研究ライン作成から最初の試行保存まで迷わず進めることを重視する。
- フォームはモバイル1カラムを基本とする。
- 必須入力は最小限にし、詳細項目は折りたたむ。
- 文字数や数値範囲の上限をUI、Zod、DBでそろえ、長文や異常値で画面が崩れないようにする。
- 「前回を複製して少し変える」導線を主要導線として扱う。

## 6. データ要件

### 6.1 データ量見積

- 試行1件あたりのテキスト情報と材料行は小さいため、v1の想定規模ではSupabase PostgreSQLの無料枠内で運用可能と見込む。
- v1では写真アップロードを扱わないため、Storage容量は設計対象外とする。
- 試行1,000件程度までを初期性能目標とし、それ以上の分析や集計はv2以降で再評価する。

### 6.2 バックアップと復元

- v1ではユーザー向けの自動バックアップ機能は提供しない。
- データエクスポートは重要だが、v1では保留し、UIにも未実装導線を置かない。実装する場合は別文書で仕様を定義してから着手する。
- 開発・運用側はSupabaseのバックアップ状況と手動エクスポート手順を確認し、[DB Migration & RLS Policy](db-migration-rls-policy.md)で扱う。

## 7. 制約・リスク

1. **スコープ膨張:** 比較、系譜、写真、公開、AIなどは魅力的だが、v1に入れると初期開発が破綻するため明示的に除外する。
2. **RLS不備:** 非公開研究ログを扱うため、RLS漏れは重大事故となる。全テーブルのRLSとテストが必須である。
3. **複製仕様:** 複製時に何をコピーするかが曖昧だとデータ不整合が起きる。試行と材料行のみをコピーし、スターはコピーしない。
4. **用語の逆戻り:** 「レシピ」「投稿」「公開」といった文言が残ると、一般的なレシピアプリへ寄る。UIでは研究ラインと試行を主語にする。
5. **非技術者前提:** プロダクトオーナーは技術的判断を行わない前提のため、実装側は危険な要望に対して反対と代替案を提示する。

## 8. 開発とテスト方針

- **初期実装:** 認証、RLS、研究ライン作成・編集・アーカイブ、試行作成・編集・論理削除、試行複製、履歴、スターを最初の対象とする。
- **UI検証:** 主要画面はモバイルとデスクトップでPlaywrightスクリーンショットを確認する。
- **RLS検証:** ユーザーAがユーザーBの研究ライン、試行、材料行、スターを読めないことを確認する。
- **スコープ管理:** v1対象外機能を実装する場合は、事前に設計文書を更新し、AI自己監査を行う。v1スコープ変更に当たる場合だけ、依頼者への限定的人間確認を必須とする。
- **Codex利用:** Codexは実装前に対象機能がv1範囲内か確認し、範囲外の場合は実装せず代替案を提示する。
- **契約文書:** MVP範囲は [MVP Scope Contract](mvp-scope-contract.md)、画面別の受け入れ基準は [Screen-by-Screen Acceptance Criteria](screen-acceptance-criteria.md) に従う。

## 9. まとめ

本要件定義書は、チャイ研究アプリのv1を、非公開の個人研究ログとして成立させるための要件を定義した。v1の核心は、研究ラインを作り、試行を記録し、前回を複製して少し変え、履歴を読み返せることである。

公開、比較、系譜、写真、AIといった拡張機能は、v1で継続利用が確認できた後に段階的に検討する。
