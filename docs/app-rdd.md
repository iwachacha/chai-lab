# チャイ研究アプリ要件定義書 (v1)

この文書は、チャイ研究アプリ(以下、本アプリ)の要件定義をまとめたものです。目的、機能要件、非機能要件、システムアーキテクチャ、データ要件、制約・リスクなどを整理し、設計・実装フェーズの指針とします。

## 1. 目的と背景

### 1.1 目的

- チャイ作りの試行錯誤を記録し、配合や工程の差と味の変化を比較することで、自分だけの理想のチャイを見つける研究ツールを提供する。
- SNS的な公開・共有機能は後期開発とし、初期バージョンは研究ログの入力・閲覧・比較に特化する。

### 1.2 背景

既存のレシピアプリや飲み物ログは完成レシピの共有や単発のチェックインを主軸にしており、チャイ愛好家が「材料・手順の微妙な違いが味にどう影響するか」を継続的に検証できる環境が存在しない。そのため、チャイ研究に特化した記録機能と差分比較機能を備えたツールが必要となった。

## 2. 用語定義

| 用語 | 定義 |
|---|---|
| **材料(material)** | 茶葉、ミルク、甘味、スパイス、生姜など、チャイを構成する単位。名称やカテゴリ、単位量などを含む。 |
| **スパイスブレンド(spice blend)** | 複数のスパイスとその量を組み合わせたプリセット。研究のたびに呼び出せる。 |
| **レシピベース(recipe base)** | 研究テーマや系統を表すベースレシピ。「朝用」「濃厚系」など。試行は必ず何らかのレシピベースを持つ。 |
| **試行(trial)** | 実際に作ったチャイ1回分の記録。使用材料、工程、評価、メモなどの情報を含む。 |
| **派生(branch)** | 試行間の親子関係。どの試行から派生して次の試行になったかを示す。 |
| **系譜ビュー** | 試行の派生関係を可視化する画面。特定のレシピベースまたは試行からの分岐を一覧できる。 |
| **お気に入り(favorite)** | 試行やレシピベースに対し、「スター」「お気に入り」「定番昇格」のステータスを付与する機能。 |

## 3. システム概要

### 3.1 ユーザーロール

- **研究者 (標準ユーザー)** … チャイを作る、試行を記録する、比較する、お気に入りに登録する。基本は単独利用で、将来の公開機能のための「公開」「限定公開」「非公開」の設定を持つ。
- **管理者 (管理UI利用者)** … システム設定、ユーザー管理、データ移行等を行う。v1では必要最低限の管理機能のみ。

### 3.2 システム構成概要

1. **フロントエンド:** Next.js ベースのPWA。
2. **ホスティング:** Cloudflare Pages Freeプラン。無制限の帯域幅、500ビルド/月および1並列ビルドまで無料。
3. **データストア:** Supabase (PostgreSQL)。Freeプランでは50k MAU、500 MB DB、1 GB Storageが提供される。RLSを活用してユーザーデータを保護する。
4. **ストレージ:** Supabase Storage を使用。保存容量が超過する場合は将来的にCloudflare R2に切り替える。R2は10GBの無料枠と無料イグレスが利用可能。
5. **認証:** Supabase Auth (Magic Linkなど)。メールアドレス認証を基本とし、将来的にOAuth追加を検討。
6. **API:** Supabase Edge FunctionsおよびNext.js API Routeで最小限のサーバーロジックを提供。将来的なAI推論API連携用にGraphQLエンドポイントも検討。

## 4. 機能要件

### 4.1 基本機能

#### FR-001 ユーザー登録・ログイン
- Magic Linkを用いたメール認証。SNSログインは将来検討。
- 2週間以内にアクティブ操作がない場合は再認証を要求する。

#### FR-002 レシピベースの作成
- ユーザーは任意の名前と説明でレシピベースを新規作成できる。
- レシピベースにタグ（例: 「朝用」「冬向け」）を付与する機能。
- レシピベースは非公開/限定公開/公開のステータスを持ち、デフォルトは非公開。

#### FR-003 試行の登録
- レシピベースを選択し、試行を作成。
- 過去の試行を1クリックで複製し、差分のみ編集できる。
- 基本入力項目: 日付、茶葉量、水量、ミルク量、甘味、主要スパイス、総合評価(5段階)、一言メモ、次回狙い。
- 詳細入力項目: スパイスごとの量・形状・投入順、煮出し時間、沸騰回数、こし方、飲んだ時間帯など。
- カスタム項目を自由に追加・編集できる。新規項目はテンプレート化し再利用可能。
- 写真は1〜3枚。各写真のサイズはCloudflare Pagesのファイルサイズ上限(25MiB)より小さく制限。
- 保存時に試行は非公開として保存される。ステータスは後から変更可能。

#### FR-004 材料プリセット・スパイスブレンド
- ユーザーは材料プリセットとスパイスブレンドを作成・編集・削除できる。
- 試行登録時にプリセットやブレンドを呼び出し、量を調整するUIを提供する。

#### FR-005 作り方・評価テンプレート
- よく使う工程(作り方)をテンプレートとして保存できる。
- 評価項目セットをテンプレートとして保存・呼び出し・編集できる。

#### FR-006 試行履歴閲覧
- 試行リストを最新順に表示。テキスト検索およびタグ・期間・評価によるフィルタリング。
- カレンダービューで日別の試行回数を確認できる。

#### FR-007 系譜ビューと派生機能
- 指定レシピベースまたは試行を起点に派生関係を木構造で表示。
- 親試行・子試行をクリックして詳細を閲覧し、新しい派生試行を作成できる。

#### FR-008 比較機能
- 任意の2試行を比較し、材料・工程・評価の差分を表示する。
- 親と子、レシピベースと試行、任意2件の組み合わせで比較可能。
- 複数試行を指定し、条件(例:「生姜5g以上」)で絞り込み、グラフ表示など簡単な統計を提供する。

#### FR-009 お気に入りと定番昇格
- ユーザーは試行に「スター」を付けられる。スターは軽いブックマーク扱い。
- 試行やレシピベースを「お気に入り」に分類し、独立したビューで一覧できる。
- 試行を「定番」に昇格すると、その時点の材料・工程・評価がレシピベースに固定され、以後ベース候補として扱う。

#### FR-010 カスタマイズ設定
- ユーザーは入力フォームの項目表示/非表示と順序をカスタマイズできる。
- 保存したカスタム項目やテンプレートは他の試行でも呼び出せる。

#### FR-011 データ出力・バックアップ
- ユーザーは自分の全データをJSON/CSVとしてエクスポートできる。
- エクスポートはローカルダウンロードのみ。後期でサードパーティ連携を検討する。

### 4.2 将来機能 (v2以降)

- 公開レシピの共有、他人のレシピの複製および派生作成。
- フォロー・コメントなどのSNS要素。
- AIによる次の実験提案、味の評価分析。
- マルチユーザーでの協働研究（グループ機能）。
- オフライン対応の向上（Service Workerで試行登録がオフラインでも可能に）。

## 5. 非機能要件

### 5.1 性能

- レスポンス時間: 主要画面表示は2秒以内を目標とする。
- 同一ユーザーの試行データ件数が5,000件程度まで性能劣化なく扱えるよう、クエリチューニングやインデックス設計を行う。
- 画像は圧縮して保存し、Cloudflare Pagesのファイルサイズ制限25MiB以下に収める。

### 5.2 スケーラビリティ

- Supabase Freeプランの制約（500MB DB、1GB storage、50k MAU、2プロジェクト、1週間非アクティブで休止）を考慮し、設計時にテーブル設計や画像圧縮を工夫する。将来的にProプランやR2移行を容易にするよう抽象化レイヤを設ける。

### 5.3 安全・セキュリティ

- SupabaseのRLSを用いて、ユーザー自身のみが試行データを閲覧・編集できる権限を設定する。公開設定が「公開」の場合のみ誰でも閲覧できるビューを設ける。
- API通信はHTTPSで行い、認証トークンをHTTP-Onlyクッキーまたはヘッダーで安全に管理。
- CSRF/XSS対策としてNext.jsの標準機構と追加Middlewareを活用する。

### 5.4 保守性・拡張性

- Next.jsおよびSupabaseのバージョンアップによる互換性を考慮し、バージョン管理と自動テストを整備。
- コア機能(試行登録・比較)と追加機能(AI提案・SNS要素)を分離したアーキテクチャとする。
- API設計はRESTfulに統一し、将来的なGraphQL拡張に備える。

### 5.5 可用性

- Cloudflare PagesはグローバルCDNで稼働し、無制限の帯域幅を活用して高い可用性を確保。
- SupabaseのFreeプランにおけるプロジェクト休止に備え、定期的なアクセスやバックアップスクリプトを運用する。

### 5.6 ユーザビリティ

- 入力負担を減らすUI (スライダー、±ボタン、プリセット選択) を採用。初期表示項目は最小限にし、詳細項目は折り畳み式。
- 系譜ビューは情報過多にならないよう、表示範囲を現在のラインに限定し、親子の差分を色分けして示す。
- ホーム画面では「前回を再現」「少し変える」という行動を中心に配置する。

## 6. データ要件

### 6.1 データ量見積

- 試行1件あたりのテキスト情報: 約3KB。5,000件で約15MB。
- 写真1枚あたり: 平均300KB(リサイズ・圧縮前提)。3枚で最大約1MB。5,000件で5GB ⇒ Supabase Storage 1GB制限を超えるので、画像アップロードは制限付きとし、別途R2を利用するか将来的に課金プランへの移行を案内する。

### 6.2 バックアップと復元

- Supabase Freeプランではバックアップが提供されない。ユーザーが手動でJSON/CSVエクスポートできる機能を提供。
- 重要なデータはオフラインエクスポートを行い、将来的なデータ移行に備える。

## 7. 制約・リスク

1. **プラットフォーム制限:** Supabase Freeプランは1週間非アクティブで休止される。アクセスが途絶えないよう、バックグラウンドで定期的にHealth Checkを行う仕組みが必要。
2. **容量制限:** DBは500MB、Storageは1GBであり、写真の数やカスタム項目が多すぎると制限超過の可能性がある。
3. **オフライン:** PWAでのローカルキャッシュは計画するが、大量データの完全オフライン対応は難しい。最低限、試行登録時の一時保存機能を実装。
4. **ユーザー教育:** カスタム項目やテンプレート機能は高度であるため、チュートリアルやヘルプを充実させないと使われない可能性がある。
5. **将来の公開機能:** 公開レシピを扱う際には著作権やプライバシーの問題が発生する可能性があり、利用規約の整備が必要。

## 8. 開発とテスト方針

- **プロトタイピング:** まず主要画面(Home・試行入力・履歴・比較・系譜)のプロトタイプをFigmaなどで作成し、5〜10名のチャイ愛好家にインタビュー。
- **イテレーション:** MVPをSupabaseのテストプロジェクトで構築し、機能単位でリリースしながらフィードバックを得る。
- **テスト:** 単体テスト（Jest/Playwright）、E2Eテスト(Cypress)を整備し、UIのレグレッションを防ぐ。
- **継続的デリバリ:** GitHub ActionsによりCloudflare Pagesへの自動デプロイを設定し、ビルド回数500/月の制限に注意しつつ運用する。

## 9. まとめ

本要件定義書は、チャイ研究アプリのv1開発に必要な仕様をまとめた。核心は「研究者が記録と比較を簡単に行えること」であり、SNS機能やAI提案は後期に回している。
