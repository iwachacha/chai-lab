# チャイ研究アプリ UI / レイアウト設計書

**作成日:** 2026-04-18

この設計書は、チャイ研究アプリの画面デザインを具体化し、AIエージェントとのペアプログラミング（Codex 想定）でも一貫した美しさと使いやすさを維持できるようにまとめたものです。色使い・フォント・グリッドシステムなどのデザインシステム、画面ごとのレイアウトパターン、フォームの設計ガイド、アクセシビリティへの配慮、そしてAIエージェントに指示するときの要点を含みます。チャイ特化アプリらしい温かみを大切にしつつ、研究ツールとしての機能性を最優先します。

## 1. デザイン方針

1. **研究ログ中心のシンプル構造** — ユーザーが記録と比較に集中できるよう、画面の情報階層を明確にします。長いフォームや複雑な設定は「質問を小さなトピック単位に分け、進むにつれて詳細を開示する」というプログレッシブ・ディスクロージャの考え方を採用します。
2. **アクセシビリティ重視のフォーム** — すべての入力フィールドには `<label>` を付け、プレースホルダで代用しないようにします。ラベルは短く、読みやすく保ちます。フィールドの長さは入力内容に合わせ、スマートフォンなどで適切なキーボードが呼び出されるように `type` 属性を設定します。入力欄は一列に並べ、複数カラムを避けることでユーザーが読み進めやすくします。
3. **エラーハンドリングとフィードバック** — フォームのエラーはインラインで表示し、どこで何が間違っているかと修正方法を明示します。主要なアクションボタンはタスクを明確に説明するテキストを使い、重要なボタンは目立たせ、二次的アクションは控えめにします。
4. **チャイらしいビジュアル** — ベージュやクリーム色を基調に、茶葉やスパイスを思わせるブラウンやグリーンをアクセントとして配します。背景やカードには紙の質感を感じる淡いテクスチャを用い、温かく落ち着いた雰囲気を演出します。これにより研究ツールでありながら、チャイを入れる時間の温かさと高揚感を表現します。
5. **AIエージェントにやさしい構成** — UIコンポーネントを明確に分割し、デザイントークン（色・余白・タイポグラフィ）を変数化してAIが容易に参照できるようにします。Codex のガイドラインでは、スクリーンショットや具体的なデザイン指示を提供し、デザインシステムのトークンや既存コンポーネントを再利用することで、より正確なUI生成ができると示されています。複数の画面状態（デスクトップ・モバイル・ホバー状態等）を提示し、Playwrightなどでの検証を併用することが推奨されています。

## 2. デザインシステム

### 2.1 カラーパレット（チャイパレット）

アプリ全体のトーンを統一するため、以下のデザイントークンを用意します。トークン名はコード内でも利用できるよう英語にし、Hex値と用途を明記します。

| トークン名 | 色コード | 主な用途 |
|---|---|---|
| `color-bg` | `#F7F2E8` | 全体の背景色。クリーム色で温かみを演出。 |
| `color-surface` | `#FFF8EF` | カードやモーダルの背景色。`color-bg`よりやや明るい。 |
| `color-primary` | `#C89D55` | メインアクション色（ボタン、リンク）。茶葉とミルクの混ざり合いを表現。 |
| `color-secondary` | `#8E5E3A` | サブアクション色。シナモンやジンジャーのブラウンを表現。 |
| `color-accent` | `#6AA68F` | アクセント色。カルダモンやミントを連想させるグリーン。ホバーやリンクに使用。 |
| `color-error` | `#C96A6A` | エラーメッセージやバッジ。控えめな赤茶色で視認性を確保。 |
| `color-success` | `#84A98C` | 成功通知やポジティブな状態表示。落ち着きのあるグリーン。 |
| `color-text` | `#3D3D3D` | メインテキスト。背景とのコントラストを確保し読みやすくする。 |
| `color-muted` | `#7B6D5D` | 補助テキストやラベル。メインより少し淡い色。 |

### 2.2 タイポグラフィ

- **本文フォント:** 日本語は游ゴシック体、英数字は Noto Sans または Montserrat。丸みのある柔らかい印象のフォントを選び、チャイの親しみやすさを表現します。
- **フォントサイズ:** ベースサイズを 16pxとし、見出しはH1:28px、H2:24px、H3:20pxとします。行間は文字サイズの1.5倍を基本とし、長文でも読みやすくします。
- **文字スタイル:** ラベルやボタンは Sentence case を基本とし、全て大文字にしないようにします。長いラベルは避け、ラベルは短くシンプルに保ちます。

### 2.3 スペーシングとグリッド

- **ベースユニット:** 4px を単位として margin・padding を設定します。上下方向は8px・16pxの倍数を使用し、要素間に一定のリズムを持たせます。
- **グリッド:** コンテンツ領域は 12 カラムを基本とし、デスクトップでは最大幅1200px、タブレットでは2カラム、モバイルでは1カラムに自動調整します。フォームは基本的に1カラムを維持し、関連項目のみ2カラムにする場合があります。
- **カードデザイン:** レシピや試行はカードとして表示し、画像、タイトル、日付、評価、ステータス（スター・お気に入り・定番）を整然と配置します。カードの角丸は8pxとし、影は控えめにします。

### 2.4 アイコンとイラスト

- **アイコン:** Heroicons や FontAwesome のラインアイコンを採用します。インプットや評価など重要なアクションには分かりやすいアイコンを使用します。サイズは24px角を基準とし、16px角や20px角も使用可能です。
- **イラスト・写真:** アプリの雰囲気を高めるために、チャイの茶葉やスパイス、ミルクをイメージしたイラストや写真を適度に取り入れます。ただし、研究ツールとしての機能性を妨げないよう、背景やヘッダー、空白スペースに控えめに配置します。

### 2.5 デザイン変数とテーマ

デザイントークンを `theme.ts` などにまとめ、色・フォント・スペーシングなどを変数化します。Codex でコンポーネントを生成する際には、このテーマファイルを参照させ、トークン名で色を指定するようにします。またダークテーマへの拡張を想定し、`color-bg-dark` などの変数も用意するとよいでしょう。

## 3. レイアウトパターンとコンポーネント

この章では、各画面や主要UIコンポーネントのレイアウトパターンを解説します。レシピ一覧や試行履歴などはカードやリスト形式で統一し、比較画面や系譜ビューは研究に特化した表現を採用します。

### 3.1 ナビゲーション

上部にアプリ名とアイコン、ユーザーメニューを配置したヘッダーを置きます。モバイルではハンバーガーメニューを使ってサイドドロワーを開き、レシピ一覧・試行履歴・設定などの主要ナビゲーションへアクセスします。ホーム画面は下部に 3 〜 5 個の固定ナビゲーションを配置することも検討します。

### 3.2 カードコンポーネント

レシピベースや試行記録はカード形式で表示します。カード内では左側に写真（1:1の比率）、右側にタイトル、日付、評価、材料サマリー、スター/お気に入りアイコンを置きます。カード全体がタップ可能で詳細画面に遷移し、二次アクション（複製、削除）はカードの右上にメニューアイコンとしてまとめます。カードは階層を表現するために背景色やシャドウの強さを変えることは避け、情報内容で区別します。

### 3.3 フォームコンポーネント

研究記録を素早く行えるよう、フォームは以下の点に留意して設計します。

- **ラベルと入力の関係を明確に** — すべての入力に `<label>` を付け、ラベルをフィールドの上に置きます。プレースホルダは補助的なヒントとし、ラベルの代用にしません。
- **一列レイアウト** — モバイル中心に考え、入力フィールドは基本的に一列に並べます。関連項目のみ 2 カラムに配置する場合でも、グループとして明確に見せます。
- **フィールド長の調整** — フィールドの長さは入力内容の長さを示唆することで、ユーザーの予想を助けます。例えば「水量 (ml)」の入力枠は短めに、「メモ」は長めのテキストエリアとします。
- **不要な項目は出さない** — 不必要な質問は非表示にし、必要な時にのみ詳細項目やカスタム項目を展開します。質問を分割し、ユーザーが負担に感じないようにします。
- **アクションボタン** — ボタンの文言は実行内容を明確に記述し、例えば「記録を保存」「複製して編集」などの具体的なテキストを用います。
- **エラー表示** — 入力エラーは各フィールドの近くに赤色のテキストとアイコンで表示し、修正方法を記載します。エラーは全角赤一色ではなく、チャイカラーに近い落ち着いた赤茶色 `color-error` を使います。
- **関連項目をグループ化** — 茶葉やミルクなどの材料を1つのセクションにまとめ、投入タイミングや量を近くに配置します。
- **プログレッシブディスクロージャ** — 詳細設定やカスタム項目はアコーディオンやモーダルにまとめ、初期画面では見せないようにします。
- **進捗インジケータ** — 長いフォームや複数ステップの入力（例えば詳細評価）では、現在のステップと残りのステップを示すステップインジケータを使用し、ユーザーがどこにいるかを示します。
- **保存と再開** — ユーザーが作業を中断できるよう、ドラフト保存機能と「続きから再開」ボタンを実装します。長い入力の途中でもデータが失われないようにします。

### 3.4 ボタンとアクション

- **Primary vs Secondary** — 主要アクションは `color-primary` を背景色にしたボタンで、無駄な境界線や影を排除し、文字色は白とします。二次アクションやキャンセルは `color-secondary` またはアウトラインボタンとして明度を下げ、誤操作を防ぎます。
- **Icon Buttons** — スターやお気に入りボタン、複製ボタンなどは20〜24pxのアイコンボタンにし、ホバー時に背景を薄い `color-accent` に変えるなどフィードバックを与えます。
- **Floating Action Button (FAB)** — モバイルでは「新規記録」のような重要なアクションをFABとして画面右下に配置することを検討します。FABは丸いボタンで `color-primary` を用い、影を付けて浮かせます。

### 3.5 リストとカレンダー

- **試行履歴リスト** — カード形式のリストとテーブル形式のリストを切り替え可能とします。カードビューは視覚的に魅力的で、テーブルビューは情報量が多い場合に便利です。フィルタリングバーを上部に置き、レシピベースや日付範囲、評価の範囲などで絞り込めるようにします。
- **カレンダー表示** — 月間/週間/日間で試行件数をアイコンや点で表示し、該当の日付をタップするとその日の試行カードへ遷移します。カレンダーは`color-primary` と `color-accent` を用いて強調します。

### 3.6 比較ビュー

比較画面では2件または複数件の試行を比較し、材料・工程・評価の差分を分かりやすく表示します。以下の設計を採用します。

- **二列デザイン** — 左右に試行Aと試行Bを配置し、各行で同じ項目を並べて差異を強調します。変更箇所は `color-accent` でハイライトし、数値の増減は矢印アイコンや色で示します。
- **差分グループ** — 材料差分、工程差分、評価差分、カスタム評価差分などのセクションに分け、ユーザーがどこが変わったかをすぐ把握できます。
- **サマリーカード** — 比較結果を要約するカードを上部に配置し、総合的な違い（評価の差、材料数の変化など）を一覧で示します。
- **アクセシビリティ** — 色だけに頼らず、増減をアイコンやテキストで示します。スクリーンリーダーでも差分内容が伝わるよう、aria属性を設定します。

### 3.7 系譜（ブランチ）ビュー

試行の派生関係を可視化する系譜ビューは、Gitのブランチ概念をヒントにしながら、チャイ研究に適した表現を取ります。

- **水平ツリー** — レシピベースや最初の試行を左側に置き、右方向に派生を伸ばします。各ノードはカード化され、試行名、日付、評価、ステータスを表示します。ノード同士を線で結び、親子関係を明確にします。
- **水平スクロール** — ノードが多くなる場合は水平方向にスクロールできるコンテナにし、閲覧中のラインだけを表示します。
- **ノード操作** — ノードをクリックすると詳細画面に遷移し、「この試行から派生」「定番に昇格」「削除」などのアクションが可能です。ノードの色やアイコンでスター・お気に入り・定番を表現し、重要度を視覚的に伝えます。

### 3.8 設定・カスタマイズ

ユーザーが自分の記録スタイルに合わせてUIをカスタマイズできるよう、以下の機能を提供します。

- **表示項目の選択と順序変更** — 基本項目は固定ですが、詳細項目やカスタム項目の表示順序をドラッグ・アンド・ドロップで入れ替えられるようにします。非表示にもでき、必要な入力のみ表示できます。
- **カスタム項目管理** — 評価軸や材料メモなどのカスタム項目を追加・編集・削除できます。カスタム項目はテンプレート化して、次回以降の試行で簡単に呼び出せるようにします。
- **テーマ切替** — ライトテーマ／ダークテーマの切り替えを提供し、色覚への配慮も考慮します。テーマ切替は`theme.ts` の色トークンを変更することで実現します。

## 4. 画面別レイアウト詳細

ここでは、要件定義書で記述した主要画面（H1, R1, R2, T1, T2, T3, T4, C1, B1, S1）のレイアウトを具体的に示します。各画面はモバイルファーストで設計し、タブレット・デスクトップではコンテンツを広く利用できるようカラム数や余白を調整します。

### 4.1 ホーム画面（H1）

- **構造:** ヘッダー（アプリ名・ユーザープロフィール）、メインコンテンツ、フッター（ボトムナビゲーション）。
- **内容:**
  - 「前回を再現」大ボタン: 直前の試行を複製して編集画面に遷移。
  - 「最近の試行」リスト: 3〜5件のカードを横スクロールで表示。
  - 「定番レシピ」グリッド: スターまたは定番昇格したレシピベースを表示し、タップで詳細へ。
  - 「研究中ライン」セクション: 各レシピベースの現在の進行状況や次のアクションを一覧。
- **ナビゲーション:** 下部に4アイコン（ホーム・履歴・レシピ・設定）を配置し、アイコンとラベルを併記して迷いを減らします。

### 4.2 レシピベース一覧（R1）

- レシピベースをカード形式で一覧表示し、検索ボックスとタグフィルターを上部に配置します。カードにはレシピ名、タグ、試行数、最終更新日を記載し、右側にメニューアイコンで編集・削除・定番昇格を行えるようにします。無限スクロールではなくページネーションや「さらに読み込む」ボタンを採用します。

### 4.3 レシピベース詳細（R2）

- **概要セクション:** レシピ名、説明、タグ、総試行数、定番昇格ボタンを表示。
- **試行タイムライン:** 新しい順に試行カードを並べ、親子関係がわかるように小さな線やアイコンで表示します。
- **系譜リンク:** 「系譜を見る」ボタンで B1 に遷移。
- **アクションバー:** レシピ編集、削除、複製ボタンを配置します。

### 4.4 試行入力フォーム（T1）

- ベース選択ドロップダウンを最上部に配置し、その下に材料追加のエリア（茶葉・ミルク・甘味・スパイスブレンド・その他）。各材料は行単位で、数量入力（数値・単位ドロップダウン）、投入タイミング（ラジオボタン）を並べます。
- 必須の基本項目（名前、日付、総合評価、次回メモ）は固定表示し、詳細項目（煮出し時間、沸騰回数、投入順序、こし方、飲んだ時間帯、狙いなど）はアコーディオンで折り畳みます。
- カスタム項目追加ボタンはセクション最後に配置し、モーダルで項目名と型を入力します。
- 下部には「保存」ボタンと「下書き保存」ボタン、キャンセルリンクを並べます。プライマリアクションは右側に配置し、色分けします。

### 4.5 試行詳細画面（T2）

- 上部に試行名、日付、レシピベース名、評価を表示し、親試行へのリンクをアイコン付きで配置します。
- **材料リスト:** テーブル風のレイアウトで材料名、量、単位、投入タイミングを表示。各材料の行を薄いセパレーターで区切ります。
- **詳細属性:** カード形式で煮出し時間、沸騰回数、投茶順序などの詳細を整理します。カスタム項目もここに表示します。
- **写真カルーセル:** 最大3枚の写真を横スワイプで閲覧できます。写真には説明文を付けられます。
- **メモ:** ユーザーが残した一言や次回に試したいことをテキストとして表示します。
- **アクション:** スター・お気に入り・定番昇格・複製・編集・削除。重要なアクションは大きなボタン、その他はアイコンにまとめます。

### 4.6 試行履歴リストとカレンダー（T3/T4）

- リスト表示ではフィルターバー（レシピベース、日付範囲、評価範囲、スター・お気に入り・定番フィルタ）を上部に固定します。各試行カードには概要情報とスター状態を表示します。
- カレンダー表示では月単位で試行数のバッジを表示し、日付をタップするとサイドパネルにその日の試行リストを表示します。週表示ではより詳細に。操作はタッチデバイスに最適化します。

### 4.7 比較ビュー（C1）

- 画面トップに試行選択バーを置き、親子関係や最近の試行候補をサジェストします。試行Aと試行Bを選択すると二列比較が展開します。差分は強調色とアイコンで示し、同じ項目は薄い色で区切り線を引いて整然と見せます。上部にサマリーカードを置き、総合評価差や材料数差を表示します。

### 4.8 系譜ビュー（B1）

- 水平スクロール可能なツリーを表示し、各ノードはクリック可能なカードです。ノードをタップするとポップオーバーまたはサイドパネルで詳細が開きます。パン・ズームジェスチャーにも対応し、スマホでも使いやすいようにします。

### 4.9 設定画面（S1）

- ユーザー情報編集（表示名、アバター変更）、カスタム項目管理、プリセット管理、テーマ切替、データエクスポート等をセクション毎にカードで分けて表示します。設定のリストはアイコンとタイトルで分かりやすく分類します。

## 5. AIエージェント（Codex）との協調のための指示

ペアプログラミングでAIエージェントにUIを実装させる際は、以下の点を明確に伝えます。

1. **明確な参考資料を用意する** — レイアウトの参考として、ワイヤーフレームやスクリーンショットを提供し、デスクトップとモバイル、ホバーやローディングなど複数状態を含めます。Codexは最小限のスクリーンショットでも実装できますが、複数の状態を提示するほど品質が上がります。
2. **デザインシステムを共有する** — `theme.ts` に定義した色・フォント・スペーシングトークンや共通コンポーネントをCodexに示し、既存のコンポーネントやデザインパターンを再利用するように指示します。プロジェクトに既にボタンやカードのコンポーネントがあれば、Codexにそれらを使うように明記します。
3. **明確かつ具体的なプロンプト** — 期待するレイアウト、コンポーネント階層、インタラクション（例えば「モーダルを使って詳細項目を開く」「カードは横にスクロールできる」）を具体的に記述します。Codexは一般的なパターンを自動的に採用しがちなので、独自スタイルやチャイらしさを出したい場合はその旨も明記します。
4. **Playwright等による検証** — Codexが生成したUIをPlaywrightなどで自動的にブラウザで比較し、スクリーンショットとの差異を判定して再度修正するサイクルを組み込みます。これにより、デザインと実装のズレを減らせます。
5. **モバイルファーストの対応** — レスポンシブデザインを重視し、モバイル・タブレット・デスクトップそれぞれのブレイクポイントでの表示をCodexに確認させます。特にフォームの幅やカードのレイアウトが変化するポイントを指定します。
6. **アクセシビリティの遵守** — ラベル、aria属性、色コントラストをCodexが正しく設定するように指示します。暗色背景でもコントラストが保てるようダークテーマ用のトークンも用意しておきます。

## 6. アクセシビリティとユーザー配慮

チャイ研究アプリは趣味の場であると同時に長時間記録する作業を伴うため、アクセシビリティとユーザーの精神的負荷に配慮します。

- **入力を小さな塊に分割** — 長い質問を小さく意味的にまとめ、一度に一つの話題に集中できるようにします。ステップインジケータで全体の進行を示し、不安を減らします。
- **保存と再開機能** — 長い作業を途中で中断できるようにし、保存できることを明示します。
- **ヘルプへのアクセス** — ユーザーが入力中に迷った場合にすぐにヘルプやガイドにアクセスできるよう、フッターやモーダルでサポートへのリンクを設けます。
- **落ち着いた言葉遣い** — エラーメッセージやガイドでは、ユーザーを責めない穏やかな文言を使います。特に研究に失敗した場合でも学びを促すような表現にします。
- **色覚多様性への配慮** — 鮮やかな赤や緑は避け、 `color-error` と `color-success` を落ち着いた色合いにし、色だけでなくアイコンやラベルで状態を識別できるようにします。

## 7. まとめ

本設計書では、チャイ研究アプリのUIとレイアウトを詳細に定義しました。研究ログを快適に記録できるようシンプルかつ温かみのあるビジュアルを採用し、フォームデザインではラベルや一列レイアウト、プログレッシブ・ディスクロージャ、エラーハンドリングなどのベストプラクティスを取り入れています。さらに、USWDSのガイドラインに沿って質問を小さく区切り、進捗表示と保存/再開機能を持たせることでユーザーの負荷を軽減します。

AIエージェントにデザインを実装させる際には、明確な参考資料とデザインシステムを提供し、既存のコンポーネントとトークンを再利用するよう指示することが重要です。Playwrightなどでビジュアル検証を行いながら、チャイらしさを損なわない美しいUIを実現してください。
