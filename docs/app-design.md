# チャイ研究アプリ UI / レイアウト設計書

**作成日:** 2026-04-18
**v1方針:** v1では、非公開の個人研究ログとして、試行を素早く記録し、前回を複製して少し変える体験に集中する。

## 1. デザイン方針

1. **研究ログ中心** — 画面の主役は完成レシピではなく、1回ごとの試行である。UI文言は「研究ライン」「試行」「前回を複製」「次回の狙い」を中心にし、一般的なレシピ共有アプリの印象に寄せない。
2. **入力の速さを優先** — v1では、ユーザーがスマートフォンで短時間に試行を保存できることを優先する。写真、公開設定、カスタム項目、複雑な比較は表示しない。
3. **前回から少し変える体験** — ホームと試行詳細では「複製して編集」を主要アクションとして扱う。
4. **モバイル1カラム** — 入力フォームはモバイル1カラムを基本とし、関連項目をまとめて表示する。
5. **装飾より可読性** — チャイらしさは色や余白で控えめに表現し、紙風カードや写真に依存しない。
6. **AI実装に強い構成** — 画面ごとの必須状態、空状態、エラー状態を明確にし、Codexが推測で画面を増やさないようにする。

## 2. デザインシステム

### 2.1 カラーパレット

| トークン名 | 色コード | 主な用途 |
|---|---|---|
| `color-bg` | `#FAFAF7` | 全体背景。白に近いニュートラルで、長時間読んでも重くしない。 |
| `color-surface` | `#FFFFFF` | フォーム、カード、モーダルの背景。 |
| `color-primary` | `#2F6F73` | 主要アクション。複製、保存など。 |
| `color-accent` | `#B7791F` | スター、軽い強調、補助的なチャイらしさ。 |
| `color-success` | `#3F7D4A` | 保存成功、復元成功。 |
| `color-warning` | `#B7791F` | 下書き、未保存状態。 |
| `color-error` | `#B94A48` | エラー表示。 |
| `color-text` | `#2F3437` | 本文。 |
| `color-muted` | `#6B7280` | 補助テキスト、説明文。 |
| `color-border` | `#D8DDD8` | 区切り線、フォーム境界。 |

チャイらしさを茶色やクリーム色の面で全面表現しない。温かさは `color-accent` と文言、材料入力の体験で控えめに出し、画面全体が茶色・ベージュ一色に見える配色は禁止する。実装では上記トークンを使い、任意の色・グラデーション・装飾的な影を増やさない。

### 2.2 タイポグラフィ

- 本文フォントはシステムUIフォントを基本とする。
- ベースサイズは16pxとする。
- モバイルでの可読性を優先し、本文の行間は1.5を目安にする。
- ボタンやラベルは短く具体的に書く。

### 2.3 スペーシングとレイアウト

- 余白は4px単位で設計する。
- フォームは1カラムを基本とする。
- 画面幅が広い場合でも、入力フォーム本文は広げすぎない。
- カードの角丸は8px以下とし、影は控えめにする。

### 2.4 アイコン

- スター、編集、複製、削除などはアイコンを併用する。
- アイコンのみのボタンには必ずラベルまたは `aria-label` を付ける。
- v1ではチャイ装飾用のイラストや写真を主要UIに使わない。

## 3. コンポーネント方針

### 3.1 ナビゲーション

- モバイルでは下部ナビゲーションを基本とし、ホーム、研究ライン、履歴、設定へ移動できるようにする。
- 主要導線は「前回を複製」と「新しい試行」に集中させる。
- 公開、共有、SNS、AI提案への導線はv1では置かない。

### 3.2 カード

- `ResearchLineCard` は研究ライン名、説明、試行数、最終試行日を表示する。
- `TrialCard` は試行名、研究ライン、日付、評価、スター状態を表示する。
- 写真領域はv1では持たない。
- カード内の二次アクションはメニューにまとめる。

### 3.3 フォーム

- すべての入力にラベルを付ける。
- プレースホルダをラベル代わりにしない。
- 材料行はカテゴリ、名称、量、単位を入力する。
- 数値入力は単位を明示する。
- 詳細項目は折りたたみ、初期状態では閉じる。
- 保存失敗時は入力内容を保持する。
- 下書き保存と破棄を明確に分ける。

### 3.4 ボタン

- 主要ボタンは「保存」「複製して編集」「新しい試行」のように動作を明確にする。
- スターは軽い印として扱い、定番やお気に入りと混同しない。
- 危険操作である削除・破棄は確認を挟む。

### 3.5 v1で作らないコンポーネント

以下はv1では作成しない。

- 比較ビュー
- 系譜グラフ
- カレンダー
- 写真カルーセル
- 公開設定トグル
- カスタム項目エディタ
- スパイスブレンド管理
- AI提案パネル
- ダークテーマ切替

## 4. 画面別レイアウト詳細

### 4.1 認証画面（A1）

- Magic Link用のメールアドレス入力欄と送信ボタンを表示する。
- SNSログイン、公開、共有、投稿を想起させる導線は表示しない。
- 送信後はメール確認を促し、認証エラー時は再送または再入力できる状態を表示する。

### 4.2 ホーム画面（H1）

- 直近の試行を表示し、「前回を複製して編集」ボタンを最も目立たせる。
- 試行が存在しない場合は、「研究ラインを作成」から始める導線を表示する。
- 最近の試行は3〜5件まで表示する。
- 写真カード、公開導線、SNS導線は表示しない。

### 4.3 研究ライン一覧（L1）

- 研究ライン名、説明、試行数、最終試行日を表示する。
- 新規作成、編集、アーカイブを提供する。
- 「レシピ投稿」「公開」などの文言は使わない。

### 4.4 研究ライン詳細（L2）

- 研究ラインの概要と、そのラインに属する試行一覧を表示する。
- 「新しい試行」ボタンを表示する。
- 試行一覧は最新順とし、スターで絞り込みできる。
- 系譜グラフは表示しない。

### 4.5 試行入力フォーム（T1）

- 基本項目は、試行名、日付、材料行、総合評価、一言メモ、次回の狙いとする。
- 材料行はカテゴリ、名称、量、単位を入力する。
- 詳細項目は煮出し時間、沸騰回数、こし方に限定し、折りたたみで表示する。
- カスタム項目、写真、公開設定は表示しない。
- 下部に「保存」「下書き保存」「破棄」を配置する。

### 4.6 試行詳細画面（T2）

- 試行名、日付、研究ライン、評価、材料行、メモ、次回の狙いを表示する。
- 親試行がある場合は「元にした試行」としてリンクを表示する。
- 主要アクションは「複製して編集」「編集」「スター」とする。
- お気に入り、定番昇格、公開、共有、写真カルーセルは表示しない。

### 4.7 試行履歴（T3）

- 最新順リストを表示する。
- 研究ライン、スター有無、日付範囲で絞り込みできる。
- カレンダー表示、グラフ、複数件比較は表示しない。

### 4.8 設定画面（S1）

- v1ではログアウトと下書き管理のみを表示する。
- データエクスポートは重要だが、v1では未実装導線を置かない。実装する場合は、エクスポート仕様と受け入れ基準を先に作成する。
- テーマ切替、項目並び替え、テンプレート管理はv2以降とする。

## 5. 状態別設計

各画面は、最低限次の状態を持つ。

- 通常状態
- ローディング状態
- 空状態
- 保存失敗状態
- 認証切れ状態

空状態では、説明文を長くしすぎず、次に取るべきアクションを1つだけ提示する。

## 6. アクセシビリティとユーザー配慮

- すべてのフォーム要素にラベルを付ける。
- 色だけで状態を示さず、テキストまたはアイコンを併用する。
- タップ対象は十分なサイズを確保する。
- エラーメッセージは入力欄の近くに表示する。
- 破棄や削除などの操作は確認を挟む。
- ユーザーを責める文言を避ける。

## 7. AIエージェント（Codex）との協調のための指示

CodexにUIを実装させる場合は、以下を守る。

1. v1対象画面以外を作らない。
2. 公開、共有、写真、比較、系譜、AI、カスタム項目のUIを追加しない。
3. デザイントークンを使い、任意の色や影を追加しない。
4. モバイル表示を優先し、フォームは1カラムで実装する。
5. 実装後、Playwrightでモバイルとデスクトップのスクリーンショットを確認する。
6. 画面の空状態と保存失敗状態を実装対象に含める。
7. 画面別の完了判断は [Screen-by-Screen Acceptance Criteria](screen-acceptance-criteria.md) に従う。
8. 未実装の将来機能を、無効ボタンや準備中表示として主要画面に置かない。

## 8. まとめ

v1のUIは、チャイ研究らしさを装飾で表現するより、試行を継続して記録できることを重視する。最初の成功条件は、ユーザーが前回の試行を複製し、少し変えて保存できることである。

比較、系譜、写真、公開、AIといった機能は、試行ログの継続利用が確認された後に段階的に追加する。
