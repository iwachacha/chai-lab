# チャイ研究アプリ企画書

## 1. 目的とコンセプト

* **目的:** チャイを作るたびに試行を記録し、前回から何を変えたか、味や満足度がどう変わったかを振り返りながら、自分に合う一杯を育てるための「個人研究ノート」アプリを作る。
* **コンセプト:** 「レシピ投稿サイト」でも「SNS」でもなく、**“理想のチャイを育てる非公開の研究ログ”**。
* **差別化ポイント:**

  * 記録は「完成レシピ」ではなく**試行単位**を主役にする。
  * ユーザーは前回の試行を複製し、変更点だけを入力して研究を続けられる。
  * v1では個人の非公開研究に集中し、公開・共有・SNS要素は扱わない。
  * 入力は固定の基本項目から始め、カスタム項目や高度な比較は利用実績を見て段階的に追加する。

## 2. 市場背景

一般的なレシピアプリは完成版の共有に重きがあり、飲み物ログ系アプリは単発のチェックインやテイスティング記録が中心である。しかしチャイ愛好家が求めるのは、「材料・工程の微妙な違いが味にどう影響するか」を継続的に検証できる環境である。

本アプリは、完成レシピを公開する場ではなく、個人が自分の試行を積み上げ、前回との差を見ながら研究を続けるための道具として位置付ける。

## 3. 採用基盤

### 3.1 フロントエンド・ホスティング

* **構成:** Next.js の静的アプリを中心としたモバイルファーストWebアプリ。
* **理由:** ブラウザで利用でき、将来的にPWA化しやすい。v1では複雑なオフライン自動同期は扱わない。
* **ホスティング:** Cloudflare Pages を採用する。

### 3.2 バックエンド

* **認証・データベース:** Supabase Auth と Supabase PostgreSQL を採用する。
* **データ保護:** Row Level Security (RLS) により、ユーザーごとの非公開研究ログを隔離する。
* **v1で扱わないもの:** Supabase Storage、Cloudflare R2、Realtime、GraphQL、外部AI APIはv1では採用しない。

### 3.3 採用方針

無料運用を前提にしつつも、無料枠を最大限使い切ることを目的にしない。v1では、機能数よりも「記録が失われない」「非公開データが漏れない」「実装が継続できる」ことを優先する。

## 4. アプリ構造とデータ設計

### 4.1 データモデル

v1では、研究ログの継続と安全な非公開保存を最優先する。材料マスター、スパイスブレンド、公開用データ、カスタム項目はMVPから外し、必要になった段階で追加する。

| テーブル/オブジェクト | 役割/主なフィールド |
|---|---|
| **research_lines** | 研究テーマを表す単位。「朝用」「濃厚系」など。ユーザーごとに非公開で管理する。 |
| **trials** | 1回の試作記録。研究ライン、親試行、日付、評価、メモ、次回の狙いを持つ。 |
| **trial_ingredients** | 試行ごとの材料行。茶葉、水、ミルク、甘味、スパイスなどをカテゴリ、名称、量、単位で保存する。 |
| **trial_stars** | 試行に付ける軽い印。v1ではお気に入りや定番昇格は扱わない。 |
| **ブラウザ内下書き** | ブラウザ内に保存する軽量下書き。DBテーブルは作らず、サーバー同期は行わない。 |

### 4.2 入力項目の考え方

1. **基本項目:** v1で常に扱う項目。研究ライン、試行名、日付、材料、総合評価、一言メモ、次回の狙い。
2. **詳細項目:** v1では最小限に留める。煮出し時間、沸騰回数、こし方など、チャイ研究に直接効く項目のみ折りたたみで扱う。
3. **カスタム項目:** v1では実装しない。将来追加する場合は、自由なJSONではなく、型・単位・表示順を持つ定義テーブルとして設計する。

### 4.3 再利用単位

v1で扱う再利用は「試行の複製」に限定する。材料プリセット、スパイスブレンド、作り方テンプレート、評価テンプレートはv2以降に回す。

## 5. UX設計

### 5.1 記録フロー

1. **初回:** 研究ラインを作成し、最初の試行を保存する。
2. **継続利用:** ホームまたは試行詳細から、前回の試行を複製して少し変える。
3. **記録:** 材料、評価、一言メモ、次回の狙いを保存する。
4. **振り返り:** 試行履歴から過去の試行を開き、親試行へのリンクで元にした記録を確認する。

### 5.2 v1で優先する体験

* スマートフォンで短時間に記録できること。
* 前回からの変更を始めやすいこと。
* 自分の研究ログが非公開で安全に守られること。
* 写真や装飾より、記録内容が読み返しやすいこと。

### 5.3 v1で優先しない体験

* 公開や共有による交流。
* 複雑なグラフや系譜ビュー。
* 任意2件比較や複数件分析。
* カスタム項目による自由入力の拡張。
* 完全なオフライン利用。

## 6. MVPとロードマップ

### 6.1 v1 MVPで必須なもの

1. Magic Linkによる認証
2. 研究ラインの作成・編集・アーカイブ
3. 試行ログの作成・編集・論理削除
4. 過去試行の複製
5. 親試行IDの保存
6. 試行履歴リスト
7. 試行へのスター付与
8. 軽量なローカル下書き保存
9. RLSによるユーザー単位のデータ隔離
10. モバイルファーストの入力UI

### 6.2 v1では実装しないもの

以下は初期開発の複雑性、セキュリティ、無料運用、実装コストの観点からv1 MVPには含めない。

* 公開・限定公開・共有URL
* SNS要素
* 写真アップロード
* カレンダービュー
* 任意2件比較、複数件比較、条件比較、グラフ表示
* 本格的な系譜グラフ
* お気に入り棚、定番昇格
* カスタム項目、評価テンプレート
* 材料プリセット、スパイスブレンド
* オフライン自動同期
* AIによる提案

### 6.3 次期バージョン以降

v1で試行ログの継続利用が確認できた後、次の順で拡張を検討する。

1. 親子2件に限定した比較表示
2. カレンダーまたは簡易統計
3. 写真アップロード
4. カスタム項目
5. 定番化した試行の管理
6. 簡易系譜ビュー
7. 公開・限定公開
8. AI提案
9. SNS要素

## 7. 技術詳細と注意点

* **Cloudflare Pages:** v1では静的アプリ配信に限定する。フルスタックNext.jsやSSRは前提にしない。
* **Supabase:** Auth、PostgreSQL、RLSを使用する。StorageやRealtimeはv1では使わない。
* **セキュリティ:** すべての業務データは非公開とし、公開・限定公開のRLSポリシーは作成しない。
* **パフォーマンス:** v1では同一ユーザーあたり試行1,000件程度まで快適に扱えることを目標とする。
* **拡張性:** 将来拡張の余地は残すが、v1のDBやUIに不要な抽象化を持ち込まない。
* **スコープ契約:** MVP範囲の最終判断は [MVP Scope Contract](mvp-scope-contract.md) に従う。

## 8. まとめ

本企画は、「チャイ研究」というニッチに振り切りつつも、最初から多機能化しない。v1では、非公開の個人研究ログとして、試行を保存し、前回を複製して少し変え、履歴を読み返せることに集中する。

比較、系譜、公開、写真、AIといった魅力的だが重い機能は、試行ログの継続利用が成立した後に段階的に追加する。初期の成功条件は、派手な機能ではなく、ユーザーが迷わず記録を続けられることである。
