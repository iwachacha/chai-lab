# チャイ研究アプリ企画書

## 1. 目的とコンセプト

* **目的:** チャイを作るたびに簡単に記録し、配合や工程の違いと味の変化を比較しながら、自分の理想の一杯を見つけるための「研究ノート」アプリを作る。
* **コンセプト:** 「レシピ投稿サイト」でも「SNS」でもなく、**“理想のチャイを育てる研究ツール”**。
* **差別化ポイント:**

  * 記録は「完成レシピ」ではなく**試行単位**が主役。
  * **複製・派生・比較・昇格**を通じて、チャイの系譜を育てる。
  * 入力は最小限から始められ、必要に応じて詳細を追加できる柔軟さ。

## 2. 市場背景

一般的なレシピアプリは完成版の共有に重きがあり、飲み物ログ系アプリは単純なチェックインやテイスティング記録が中心である。しかしチャイ愛好家が求めるのは、「材料・工程の微妙な違いが味にどう影響するか」を継続的に検証できるツールである。既存サービスにはこのニッチに特化したものがなく、**チャイ研究に特化したUXを提供する余地**が大きい。

## 3. 採用基盤

### 3.1 フロントエンド・ホスティング

* **構成:** Next.js (静的ページとクライアントサイド中心のPWA)
* **理由:** PWAならモバイルにもインストール可能で、App Store/Google Play審査を経ずに配布できる。
* **ホスティング:** Cloudflare Pages を採用。特徴は以下。

  * Freeプランでも**無制限の帯域幅**を提供。
  * 300以上のエッジロケーションにより高速配信が可能。
  * 月500回のビルドと1回の同時ビルドが無料枠。
  * プロジェクトごとに100個のカスタムドメイン、20,000ファイルまで許容。

### 3.2 バックエンド

* **データベース & 認証:** Supabase

  * Freeプランで**50,000 MAU、500MBデータベース、1GBファイルストレージ、5GB egress**を提供。
  * 無制限のAPIリクエストと基礎的なRealtime/Edge Functionsが含まれる。
  * 1週間アクティビティが無いとプロジェクトが一時停止、最大2プロジェクト。
  * Row Level Security (RLS) やPostgreSQLベースのクエリが使えるため、公開・非公開データの分離が容易。
* **ストレージ拡張:** Supabase Storageの1GB制限を越えたらCloudflare R2への移行を検討。R2は無料枠が10GBかつ出力（イグレス）が無料。

### 3.3 代替案の考察

* **Appwrite:** 75,000 MAU、2GBストレージ、750K関数実行などのFreeプランがあるが、プロジェクト数が2個に制限され、1週間未使用で一時停止になる。チャイ研究向けに必要な柔軟性とコミュニティ事例の豊富さを考慮し、Supabaseを優先。
* **Firebase:** Authentication の無料枠が3,000DAU、有償課金でCloud Storageへの依存度が増すなど制約が大きい（従来の議論より省略）。

## 4. アプリ構造とデータ設計

### 4.1 データモデル

以下は主要テーブル(概念)と重要フィールドの例。実際は更に細かい属性を適宜拡張する。

| テーブル/オブジェクト       | 役割/主なフィールド                                                          |
| ----------------- | ------------------------------------------------------------------- |
| **materials**     | 茶葉・ミルク・甘味・スパイスなどの基本材料。名称、カテゴリ、単位など。                                 |
| **spice_blends**  | よく使うスパイスの組み合わせ。スパイスIDと量のリスト。                                        |
| **recipe_bases**  | 「朝用」「濃厚系」など研究テーマ。名称、説明、作者ID。                                        |
| **trials**        | 1回の試作記録。recipe_base_id、派生元trial_id、材料リスト、工程、評価、日時、メモ。               |
| **branches (系譜)** | 派生関係を管理。parent_trial_id、child_trial_id。                             |
| **favorites**     | 気に入った試行やレシピベースを昇格させる。trial_idまたはrecipe_base_id、ステータス(スター／お気に入り／定番)。 |

### 4.2 入力項目の3層構造

1. **基本項目:** 誰もが毎回入力する最小限（レシピ名、日付、茶葉量、水量、ミルク量、甘味、主要スパイス、総合評価、次回メモ）。
2. **詳細項目:** 研究モードで追加表示する。スパイスごとの投入タイミング、形状、煮出し時間、沸騰回数、こし方、飲んだ時間帯、狙いなど。
3. **カスタム項目:** ユーザーが自由に作成・再利用できる評価項目やメモ欄（例: 「チャイ感」「目覚め効果」「雨の日向け」）。

### 4.3 再利用単位

* **材料プリセット:** よく使う茶葉・ミルク・甘味・スパイスのセット。
* **スパイスブレンド:** 「生姜強め」「カルダモン主役」などの組み合わせ。
* **作り方テンプレ:** 「水先行煮出し」「ミルク多め短時間」など工程の型。
* **評価テンプレ:** 「普段用」「生姜研究用」など比較用スコアシート。

### 4.4 派生・系譜機能

* **1タップ複製:** 任意の試行(trial)を複製し、差分のみ入力する。
* **系譜ビュー:** Gitのブランチ概念を応用し、親子関係を線で可視化。親との差分やベースとの比較をワンタップで表示。大量のノードを表示する全体グラフではなく、閲覧中のラインに絞ったローカル系譜を採用。
* **比較機能:** 材料・工程・評価の差分を一覧表示。親子比較、任意の2試行比較、条件絞り込み比較(例えば「生姜7g以上」)が可能。

### 4.5 お気に入り・定番昇格

* **スター:** 気に入った試行に軽く印を付ける。
* **お気に入り:** より深く見返したい試行やベースをまとめる棚。
* **定番昇格:** レシピベースとして固定し、次回以降のベース候補に格上げ。

## 5. UX設計

### 5.1 記録フロー

1. **ホーム:** 前回の続きから開始、最近の試行一覧、定番レシピ、研究中のラインを表示。「前回を再現」「少し変える」ボタンを大きく配置。
2. **新規試行:** ベースを選択→複製→変更点のみ編集。数値入力は ± ボタンやスライダーで直感的に調整。写真は1〜3枚。
3. **評価:** 軽い項目(総合満足度)のあとに詳細項目へ遷移。最後に「次回の狙い」欄を必須にし、記録を次回への橋渡しにする。
4. **詳細モード:** スワイプやトグルで詳細項目を開閉。初めてのユーザーを混乱させない。

### 5.2 比較・振り返り

* **比較画面:** 材料差分、工程差分、評価差分を色分け表示。2件比較、ベースとの差分、条件比較がワンタップ。
* **研究ノート画面:** カレンダー・タイムライン・フィルタで研究履歴を探索。「失敗」や「成功」など評価タグで絞り込み。
* **系譜画面:** 派生ラインを枝状に表示。任意のノードから分岐させたり定番に昇格したりできる。

### 5.3 カスタマイズ

* **項目の表示・非表示・並び替え:** 詳細項目とカスタム項目はユーザーが並び替えられる。基本項目は固定。
* **テンプレ化:** 作成したカスタム項目・材料プリセット・スパイスブレンドをテンプレとして保存し、次回から選択可。

## 6. MVPとロードマップ

### 6.1 MVPで必須なもの

1. ベース作成・複製・試行ログ保存
2. 基本入力フォーム＋詳細項目へのアクセス
3. 1タップ複製と差分入力
4. 評価入力と「次回メモ」
5. 試行履歴（リスト＆カレンダー）
6. 簡易比較（親とベース）
7. お気に入り（スター→定番昇格）
8. Supabase Auth（メール or Magic Link）とRLSによるデータ隔離
9. PWAインストール対応（Service Worker、オフラインキャッシュ）

### 6.2 次期バージョン以降

* 条件比較や複数件同時比較の強化
* 系譜ビューの充実（折れ線やタグのフィルタ）
* AIによる「次の一手」提案（過去データから予測）
* 公開・共有機能（限定公開→一般公開へ段階的拡張）
* SNS的要素（フォローやコメント）は成功後の後期フェーズで検討

## 7. 技術詳細と注意点

* **Cloudflare Pages:** 無料プランは月500ビルド・1同時ビルド。ビルドは20分でタイムアウトし、ファイルサイズは25MiBまで。プロジェクトは100件まで。PWAの静的アセットは20,000ファイル以内に収める。
* **Supabase:** Freeプランは50K MAU・500MB DB・1GBファイル保存・5GB egress。アクティブプロジェクトは2個まで、1週間非アクティブで休止。長期運用や画像増加に備えて、必要に応じて有料プランやCloudflare R2への移行を見込む。
* **セキュリティ:** RLSでユーザーごとに試行データを隔離し、「公開」「非公開」「限定公開」のステータスに応じて条件付アクセスを設定。公開データのみ静的サイトとして生成できる構成も検討。
* **パフォーマンス:** 材料や工程が肥大化してもレスポンスを保つため、非正規化ビューやキャッシュテーブルを用意。比較機能では差分計算にSQLのJSON/BINARYデータ型を活用する。
* **拡張性:** ベーステーブル設計を柔軟にし、将来スパイスの属性や評価項目の追加が容易にできるよう、スキーマの一部をJSONBにする等の工夫を検討。

## 8. まとめ

本企画は「チャイ研究」というニッチに振り切り、完成レシピではなく**研究プロセス**を主役に据えることで、既存サービスにはない体験を提供する。Cloudflare PagesとSupabaseの組み合わせにより、**無制限の帯域幅**と**50K MAU／500MB DB**という強力な無料枠を活用しつつ、データを安全に管理できる。入力フローを軽量化し、複製・派生・比較・昇格の仕組みで使いやすさと研究価値を両立させることが、本プロジェクトの成功の鍵である。
