# チャイ研究アプリ プロジェクト方針

**作成日:** 2026-04-18

本ドキュメントは、チャイ研究アプリの方向性と開発運営の基本方針を示す。v1では、非公開の個人研究ログとして、研究ライン、試行、複製、履歴、スターを安全に実装することを最優先する。

## 1. ビジョンと目的

チャイ研究アプリの目的は、チャイ愛好家が自分の試行を継続的に記録し、前回から少し変えながら理想の一杯を探せるよう支援することである。

このアプリは、単なるレシピ共有ツールでもSNSでもない。v1では公開・共有・AI提案を扱わず、個人の非公開研究ノートとして成立させる。

## 2. 基本理念

### 2.1 研究ログ中心

すべての機能は、試行を記録し、複製し、読み返す体験を支えるために存在する。完成レシピや投稿を中心にせず、1回ごとの試行を主役にする。

### 2.2 入力負荷の最小化

フォームはモバイル1カラムを基本とし、必須入力を最小限にする。詳細項目は折りたたみ、写真やカスタム項目などの重い入力はv1では扱わない。

### 2.3 データの安全性とプライバシー

v1の全データは非公開とする。SupabaseとPostgreSQLを活用し、RLSを徹底することで、ユーザー自身の研究ログのみを閲覧・編集できるようにする。公開・限定公開の設計はv2以降で別途検討する。

### 2.4 段階的な拡張

比較、系譜、写真、カスタム項目、AI、SNSは、v1で試行ログの継続利用が確認できた後に段階的に検討する。将来拡張の余地は残すが、v1のDBやUIに不要な複雑性を持ち込まない。

### 2.5 継続的な改善

設計書は固定された最終形ではなく、実装と利用フィードバックを通じて更新する。ただし、スコープ変更、DB変更、RLS変更、外部サービス連携は、設計文書を更新してから実装する。

## 3. 開発プロセス

### 3.1 MVP優先

v1 MVPでは、認証、研究ライン、試行、材料行、複製、履歴、スター、軽量下書き、RLSに集中する。範囲外機能は実装しない。MVP範囲の最終判断は [MVP Scope Contract](mvp-scope-contract.md) に従う。

### 3.2 ユーザーテストとフィードバック

初期テストでは、次の点を確認する。

- 初回ユーザーが研究ラインを作成できるか。
- 試行を短時間で保存できるか。
- 前回の試行を複製して変更できるか。
- 保存した試行を履歴から見返せるか。

比較や系譜などの高度機能は、上記の基本体験が成立した後に検証する。

### 3.3 コード品質と自動テスト

ESLint、Prettier、TypeScript strictで基本品質を保つ。Vitest、React Testing Library、Playwrightを使用し、フォーム、主要導線、RLSの検証を行う。Playwrightではモバイルとデスクトップの主要画面を確認する。

### 3.4 AIエージェントとの協調開発

Codexや他のAIエージェントは、ユーザーの指示をそのまま実装するのではなく、品質・安全性・運用性を守る開発パートナーとして振る舞う。

特に以下の場合、Codexは実装前に反対理由と代替案を提示する。

- MVP範囲外の機能を求められた場合
- 公開、共有、AI連携、写真保存、オフライン同期など、セキュリティや運用負荷が大きい機能を求められた場合
- DBスキーマやRLSに影響する変更がある場合
- 新しいライブラリの追加が必要な場合
- 既存のUI方針から外れ、一般的なレシピアプリやSNSに寄る場合

Codexは、DB変更を行う場合はmigration、RLS、テスト観点をセットで提示する。UI変更を行う場合は、モバイルとデスクトップの主要画面をPlaywrightで確認する。MVP外機能は、明示的な設計判断なしに実装してはならない。詳細な実行ルールは [Codex Execution Rules](codex-execution-rules.md) に従う。

## 4. デザインとユーザー体験のガイド

### 4.1 チャイらしさの扱い

チャイらしさは、色、余白、文言、材料入力の体験で控えめに表現する。写真や紙風装飾に依存せず、研究ログとしての読みやすさと入力しやすさを優先する。

### 4.2 用語

UIでは「研究ライン」「試行」「元にした試行」「次回の狙い」を中心に使う。「投稿」「公開レシピ」「フォロー」「コメント」など、SNSや一般的レシピアプリを想起させる用語はv1では使わない。

### 4.3 フィードバック

保存、下書き、破棄、削除などの状態は明確に伝える。エラーは入力欄の近くに表示し、ユーザーを責めない文言にする。

## 6. ドキュメントとコミュニケーション

### 6.1 ドキュメント更新

要件定義、詳細設計、UI設計、技術スタック、プロジェクト方針は、実装の判断基準として扱う。実装で仕様変更が必要になった場合は、該当文書を更新してから作業する。

### 6.2 ベンダー関係と役割分担

本プロジェクトでは、プロダクトオーナーであるユーザーが技術的な知識を持たないことを前提に要望や意見を伝える。一方、AIエージェントを含む開発側は、開発ベンダーとして振る舞い、ユーザーの指示をただ実装するのではなく、専門家として実現可能性、安全性、保守性、既存設計との整合を吟味する。

要望に技術的な矛盾や破綻が含まれる場合、既存機能と安全に統合できない場合、セキュリティや性能に悪影響がある場合には、開発側は率直に反対意見を述べ、代替案や段階的な実現案を提示する。ユーザーの意図は尊重するが、品質と安全性を損なう実装は行わない。

## 7. おわりに

チャイ研究アプリのv1は、多機能なレシピサービスを目指さない。まず、ユーザーが自分の試行を安全に記録し、前回を複製して少し変え、履歴を読み返せる状態を作る。

この基礎が成立してから、比較、系譜、写真、公開、AIなどを段階的に検討する。本方針は、プロジェクトを小さく始め、品質と継続運用性を守るための基準である。
