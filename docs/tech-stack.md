# チャイ研究アプリ 技術スタック候補一覧

**作成日:** 2026-04-18

本書では、要件定義書・詳細設計書・UI設計書を踏まえて、チャイ研究アプリに適したフロントエンド技術スタックと外部ツールの有力候補を整理します。インフラ（Cloudflare PagesとSupabase）は既に確定しているため、それ以外のライブラリやOSS、開発支援ツールについてカテゴリ別にまとめ、将来の拡張性やAIエージェント（Codex）との協調を考慮した選択肢を提示します。すべてを導入する必要はなく、実際の開発状況やチームの好みに応じて採用・置き換えを検討してください。

## 1. UIフレームワークとデザインシステム

### 1.1 UIコンポーネントライブラリ

| ライブラリ | 特徴 | 採用メリット | 採用リスク |
| --- | --- | --- | --- |
| **Chakra UI** | React向けのアクセシブルなUIコンポーネント集。テーマ機構とスタイル Props を提供し、カスタムテーマにチャイパレットを設定しやすい。 | デザインシステムとの親和性が高く、色・余白などをトークン化しやすい。アクセシビリティ対応が組み込まれており、フォームのラベルやエラー表示などのベストプラクティスを簡単に実装できる。 | コンポーネントが多数用意されているが、細かなカスタマイズを行う場合はスタイル Props が冗長になることがある。 |
| **Tailwind CSS + Headless UI** | ユーティリティファーストのCSSフレームワーク。Headless UIと組み合わせることでアクセシブルな対話コンポーネントが構築できる。 | クラスを組み合わせて柔軟にレイアウトを作れるため、チャイらしいテイストを表現しやすい。ダークモードやテーマの切替も容易。 | クラスの冗長化や命名の煩雑さがあり、開発初期にはデザインガイドラインを明確にしておく必要がある。 |
| **Material UI (MUI)** | GoogleのMaterial Designに準拠したコンポーネント集。豊富なコンポーネントとレイアウトシステムを提供。 | 入力コントロールやステップインジケータなどの複雑なコンポーネントをすぐ使える。 | Material Designの雰囲気が強く、チャイらしいオリジナリティを出すにはカスタムテーマの作り込みが必要。 |

### 1.2 スタイル管理

- **CSS-in-JS (Emotion / styled-components):** コンポーネント単位でスタイルを記述し、チャイパレットやグリッドサイズなどのテーマ変数を簡単に利用できる。TailwindやChakraと組み合わせて使う場合もある。
- **Design Tokens:** カラーパレットやタイポグラフィ、スペーシングを JSON または TS ファイルで定義し、アプリ全体で参照する。Codexに提示することでAI生成UIでも一貫性を保てる。
- **Storybook:** UIコンポーネントの開発とドキュメンテーションに便利。コンポーネントを単体で確認でき、CodexにFigmaやスクリーンショットを提供する際の参考にもなる。

## 2. 状態管理とデータ取得

| ライブラリ | 特徴 | 採用理由 |
| --- | --- | --- |
| **React Context + Zustand** | 軽量な状態管理。ユーザーセッションやUI状態の管理に適する。 | グローバル状態管理が必要な場面が限られているため、簡潔に書ける。
| **TanStack Query (旧 React Query)** | データ取得とキャッシュ管理に特化。Supabaseや他のAPIからデータを取得し、オフライン時のキャッシュ利用や再取得を管理できる。 | APIの更新時にキャッシュを自動で更新し、UIの同期を保つ。オフラインファーストの実装に役立つ。
| **Zod + React Hook Form** | スキーマベースのフォームバリデーション。React Hook Formはパフォーマンスに優れ、大規模フォームでも高速。 | 入力値の検証とエラーメッセージを簡単に表示でき、インラインエラー表示や必須項目チェックをベストプラクティスに沿って実装できる。 |

## 3. グラフ・比較・可視化

| 項目 | 候補ライブラリ | 説明 |
| --- | --- | --- |
| **チャイ指紋の可視化** | **Radar Chart:** Recharts, Chart.js, echarts-for-react | 評価軸をレーダーチャートで示す場合に使用。色数を抑え、チャイパレットに合わせたカスタムテーマを適用する。 |
| **比較差分の表示** | **Diffビュー:** react-diff-viewer, diff2html | 材料や工程の差分を見せる際に、2カラム表示と差分ハイライトを提供。デフォルトの色使いを `color-accent` に変更する必要あり。 |
| **系譜ツリー** | **React Flow**, D3.js, **Graphviz via viz.js** | ブランチ構造の可視化に適したライブラリ。React Flowはドラッグ操作やノードカスタム表示が容易で、木構造や複雑な連結関係の表示に向いている。 |

## 4. オフライン機能とPWA

| ライブラリ / ツール | 用途 | コメント |
| --- | --- | --- |
| **Workbox** | Service Worker生成とキャッシュ戦略管理。App ShellはCache First、APIはNetwork First、画像はStale-While-Revalidate戦略を簡単に適用できる。 | Cloudflare PagesにデプロイするPWAで必須。 |

| **Next-PWA** | Next.js向けのPWAプラグイン。manifest生成やSW登録を自動化。 | プロジェクトのセットアップが簡単になる。 |

| ツール | 用途 | コメント |

## 7. Codex連携に関するツールとワークフロー

### 7.1 Codexプロンプトガイドライン

CodexでUIコードを生成させる際は、次のポイントに留意してプロンプトを作成します。

1. **明確な参照画像やワイヤーフレームを添付** — 画面の完成イメージや各状態（モバイル/デスクトップ、ホバー、空状態）を提供し、Codexが推測せずにUIを再現できるようにします。
2. **デザインシステムの指定** — カラーパレット・タイポグラフィ・スペーシングトークン、既存のボタンやカードコンポーネントのパスをプロンプト内で説明し、再利用を促します。
3. **具体的なレイアウト・インタラクションの指示** — 例として「材料入力は1カラムで表示」「アコーディオンで詳細項目を展開」「ノード同士を線で結ぶ水平ツリー」「カレンダーでタップするとサイドパネルを開く」など、動作と配置を明示します。
4. **検証と修正サイクル** — Codexが生成したUIをPlaywrightやCypressで検証し、スクリーンショットとの差異を具体的にフィードバックするワークフローをプロンプトに含めます。差異がある場合は、「デザインシステムのトークンを優先して必要最小限の調整を行う」と指示します。
5. **アクセシビリティ要件の明示** — `aria-label` の使用、ラベルと入力の対応、インラインエラー表示、色覚多様性への配慮などを説明します。

### 7.2 Codex Skills とプラグイン

| Skill / プラグイン | 用途 | コメント |
| --- | --- | --- |
| **Codex + Playwright** | UIの自動検証。Codexが生成した画面を Playwright で実行し、スクリーンショットを取得して比較。 | Codex use case ドキュメントでも推奨されている。 |
| **Codex + Figma** | Figmaからコンポーネントやデザイン情報を取得し、コードに反映。 | UIデザイナーが作成したFigmaファイルと開発を同期させる。 |
| **Codex Design Tokens** | 独自のデザイントークンを Codex に認識させ、出力時に適切な色やフォントを適用させる。 | テーマ変更時に素早く対応できる。 |
| **$playwright-interactive** | 対話的にブラウザを開き、UIを実際に操作して検証。 | 明細修正のループを高速化。 |

## 8. その他の有用OSS・参考資料

- **date-fns / Day.js** — 日付処理を軽量に行い、試行日時の表示や比較に便利。
- **Lodash / Ramda** — データ変換や比較ロジックを簡潔に記述。差分計算やフィルタ処理に使用。
- **Dexie.js** — IndexedDBラッパーとしてオフラインデータの保存と同期に適している。Supabaseへの同期に失敗した場合もローカルに保持できる。
- **React DnD** — 設定画面の項目並び替えなどドラッグ＆ドロップの実装に使用。
- **React Icons** — さまざまなアイコンライブラリの統合パッケージ。HeroiconsやFontAwesomeをまとめて使える。
- **Form validation examples** — NNGroupやW3Cのフォームガイドラインでは、ラベルを付け、プレースホルダをラベルとして使わない、1カラムレイアウトを推奨、エラーはインラインで明示するなど、今回のフォーム実装で参考になる。USWDSの「Complete a complex form」パターンでは質問を小さな単位に分け、進捗を示し、保存と再開を提供することが推奨されている。

## 9. まとめ

チャイ研究アプリの技術スタックは、React / Next.js を中心に、多くの優れたOSSを組み合わせることで柔軟かつ拡張性の高いものとなります。UIコンポーネントライブラリには Chakra UI や Tailwind CSS + Headless UI を候補とし、フォームには React Hook Form と Zod、状態管理には TanStack Query を推奨します。PWAとオフライン対応には Workbox と Dexie.js を活用し、チャートや系譜表示には Recharts や React Flow などの可視化ライブラリを採用するのが良いでしょう。

Codexを用いたペアプログラミングを行う場合、明確なデザインリファレンスとデザイントークンを準備し、既存のデザインシステムを再利用するよう指示することが成功の鍵です。さらに、PlaywrightなどによるUI検証をプロンプトに組み込むことで、AIが生成したコードと期待するデザインの差異を埋める仕組みを整えます。
