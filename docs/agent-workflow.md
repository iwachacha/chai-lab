# Agent Workflow

**作成日:** 2026-04-19

この文書は、AIエージェントがチャイ研究アプリPJを実装・修正するときの標準手順を定義する。役割と判断権限は `docs/agent-relationship-governance.md` を優先し、この文書は実際の作業順、停止条件、証跡を扱う。

このworkflowはプロジェクト共通の作業手順として扱う。現時点の判断基準はv1であり、v1固有の対象外機能やDB/UI境界は `docs/mvp-scope-contract.md` と各契約文書に従う。将来フェーズへ進む場合は、このworkflowを作り直すのではなく、フェーズ固有のスコープ文書と参照先を更新する。

## 1. 適用範囲

このworkflowは、次の作業に適用する。

- 実装開始前の判断
- DB/RLS/RPC/Data Access/認可境界の変更
- 現行フェーズスコープに触れる要望
- 停止、反論、代替案提示
- 依頼者へのエスカレーション
- 作業記録と未実施検証の記録

現行フェーズのスコープ、画面仕様、DB契約そのものはこのworkflowでは変更しない。具体的な許可機能、禁止機能、成立条件は `docs/INDEX.md` の現行フェーズ契約を参照する。

## 2. 通常実装workflow

### 開始条件

- 依頼が現行フェーズ範囲内、または現行フェーズ範囲内か判断できる情報がある。
- 作業対象の参照文書が特定できている。

### 手順

1. 今回の確認対象が `ローカル作業ツリー`、`直近コミット`、`公開 repo / 既定ブランチ` のどれかを宣言し、混在させる場合は用途を切り分ける。
2. `docs/INDEX.md` で文書分類を確認し、現行フェーズ契約で範囲内か確認する。
3. 影響レイヤーを分類する。分類は UI、Data Access、DB/RLS/RPC、Auth、Deploy、Docs、Test のいずれかに置く。
4. 対象レイヤーの契約文書を読み、今回の正本ファイルと派生文書の従属関係を固定する。
5. AIが採用方針を1つに決める。複数案を残したまま依頼者へ技術判断を渡さない。
6. 正本ファイルを先に修正する。worklogや要約文を先に完成させない。
7. 関連文書、template、必要な既存worklog、補助文書、関連参照を正本解釈へ収束させる。
8. 変更種別に応じた検証を実行し、検索、差分、コマンド結果で旧解釈、旧様式、古い停止条件の残存を確認する。実行できない場合は未実施記録を残す。
9. 作業記録には、repo上で成立した事実だけを書く。未解消事項や停止条件は消さず、未確認のまま残す。
10. 変更がある場合は push し、GitHub 上の対象ブランチとコミット識別情報を確認する。
11. CI確認が必要な変更では push 後にCI結果まで確認し、不要な場合も要否判断と理由を記録する。

### 変更規模の分類、記録密度、完了条件

手順6の検証前に、作業を「軽微変更」または「大きなコード変更」に分類する。分類は行数やファイル数だけで固定せず、実行時挙動、影響レイヤー、失敗時の影響、必要な検証量で判断する。判断が揺れる場合は、変更の失敗がユーザー操作、保存データ、認可、安全性、リリース可否へ波及するかを優先して見る。

危険変更workflowは、変更そのものが migration、RLS、grant/revoke、`security definer`、RPC、認可境界、Auth Redirect、env境界、静的export制約へ触れる場合に適用する。これらを説明するdocs-onlyの整理やtemplate更新は、対象実装や停止条件を弱めない限り通常実装workflowで扱える。ただし、危険変更の必須監査や停止条件を薄める文言変更は行わない。

#### 軽微変更

次のように、影響が局所的で検証コストに比してリスクが明らかに小さい変更は軽微変更として扱える。

- docs-onlyの追記、表記修正、リンク補正、索引やテンプレートの小修正
- コメント、文言、型名の誤記など、実行時挙動を変えない修正
- 対象と影響が明確な小さな設定修正
- 既存テストや機械的確認で十分に確認できる局所的なスタイル、文言、非本質的な修正

軽微変更では、変更種別に応じた最低限の確認を行う。docs-onlyなら `npm run check:docs`、設定変更なら該当するlint/typecheck/buildなど、既存ルールが求める確認は省略しない。軽微変更でも、変更を伴う場合は完了報告前に必ずpushしてGitHub反映を確認する。CI確認は変更内容と既存運用で要否判断し、未確認なら理由を残す。

軽微変更の完了条件は次とする。

1. 変更が局所的で、危険変更workflowの対象実装や大きなコード変更へ広がっていない。
2. 変更種別に応じた最低限の確認を実行している。
3. worklogまたはPR本文には、全変更共通の必須項目だけを最小セットとして残している。
4. push済みで、GitHub上の対象ブランチとコミット識別情報を確認している。
5. CI確認の要否判断を記録している。

#### 大きなコード変更

次のいずれかに当たる場合は、大きなコード変更として扱う。

- 複数ファイルまたは複数レイヤーにまたがる実装変更
- ロジック、バリデーション、画面挙動、状態管理、ルーティング、Data Access、Auth、Deployの変更
- DB/RLS/RPC、migration、権限境界、env境界、静的export制約に触れる変更
- 既存テストの追加・更新が必要になる変更、または手動の動作確認なしでは成立判断が弱い変更
- 失敗時にユーザー操作、保存データ、認可、安全性、リリース可否へ影響する変更

大きなコード変更は、次を満たすまで完了扱いにしない。

1. 変更内容に応じたローカルのテストまたは検証を実行する。
2. 必要な動作確認を行う。UIなら主要ビューポート、Data AccessやDB/RLS/RPCなら成功/失敗/拒否条件など、対象契約に沿って確認する。
3. worklogまたはPR本文に、全変更共通の必須項目に加えて追加項目まで残す。
4. 変更をpushする。
5. GitHub上の対象ブランチとコミット識別情報を確認する。
6. push後のCI結果を確認する。
7. CIが失敗した場合は、原因に応じて修正し、必要なローカル検証を再実行し、再pushしてCI成功まで確認する。
8. その場で閉じられない場合は、未解決理由、失敗内容、残タスク、次に止める条件を記録し、成功したかのように完了報告しない。

PRがある場合はPR本文、PR前またはAI単独作業では作業記録へ、分類、実施検証、動作確認、push/CI結果、未完了なら残タスクを残す。危険変更workflowに該当する場合は、大きなコード変更の追加項目に加えて対象専用の監査項目も残す。

#### GitHub反映とCI要否判断

変更を伴う作業は、docs-only、設定変更、小修正、コード変更を問わず、ローカルだけで完了扱いにしない。完了報告前に必ず次を満たす。

1. push済みである。
2. GitHub上の対象ブランチとコミット識別情報を確認できる。
3. 作業記録またはPR本文に、今回の判断根拠が `ローカル作業ツリー`、`直近コミット`、`公開 repo / 既定ブランチ` のどれかを明記している。

CI確認は次の場合に必須とする。

- 大きなコード変更
- `.github/workflows/**`、`scripts/check-operational-docs.mjs`、`package.json` の検証コマンド、deploy / release経路を変更した場合
- 依頼者が明示的にCI確認を求めた場合

それ以外では、CIを未確認にできる。ただし、要否判断、未確認理由、代替確認、残リスクを記録する。

### 停止条件

- 現行フェーズ対象外機能が必要になる。
- DB/RLS/UI/受け入れ基準のいずれかが欠けた機能実装になる。
- 危険変更workflowの対象なのにAI自己監査資料を作れない。
- 実Production、secret、本番データ、外部サービス契約に触れる。

## 3. 危険変更workflow

### 対象

次に触れる場合は、通常実装workflowに加えてこのworkflowを必ず通す。

- migration追加/変更
- RLS policy追加/変更
- grant/revoke変更
- `security definer` helper/RPC
- RPC追加/変更
- DBスキーマ制約変更
- 認可境界に関わるData Access変更
- AppError契約変更
- Auth Redirect、env境界、静的export制約

### 手順

1. 変更の目的と現行フェーズスコープ根拠を書く。
2. 影響範囲を列挙する。最低限、table、column、index、policy、function、RPC、grant/revoke、Data Access、UI、testを確認する。
3. 代替案を比較し、安全、単純、可逆、監査可能な案を1つ採用する。
4. RLS/policy matrixを作る。role、操作、`USING`、`WITH CHECK`、拒否条件を明確にする。
5. grant/revoke一覧を作る。anon、authenticated、PUBLICの許可と拒否を明確にする。
6. `security definer` を使う場合は、`search_path` 固定、`auth.uid()`、所有者確認、PUBLIC revoke、必要roleへのgrantを満たす。
7. RPCを使う場合は、入力、認可、戻り値、失敗時挙動、AppError分類、部分保存なしを定義する。
8. direct CRUD拒否を確認する。`trials` と `trial_ingredients` の書き込みは定義済みRPC以外から成立させない。
9. A/B分離、anon拒否、想定外操作拒否、想定経路成功を検証する。
10. AI自己監査記録を残す。

### 完了条件

- 採用方針、代替案比較、影響範囲、権限境界、検証結果、未実施検証、残リスク、停止範囲が記録されている。
- 現行フェーズ非対象の列、テーブル、UI導線、依存が混入していない。
- 必須検証が成功している。未実施がある場合は、該当依存タスクを完了扱いにしていない。

### 局所停止条件

- RLS、grant/revoke、A/B分離、anon拒否、direct CRUD拒否のいずれかが未検証。
- `security definer` のhardeningが未確認。
- RPCのAppError分類や部分保存なしが未確認。
- migrationが複数論理変更を混在している。
- スコープ外の将来機能が混入している。

## 4. 反論・代替案workflow

### 開始条件

依頼または実装案が次に該当する。

- 現行フェーズスコープ外
- RLS弱体化、権限事故、direct CRUD許可につながる
- DB/RPC/Data Access責務が混線する
- 静的export制約と衝突する
- 将来機能の先回りになる
- 不要なライブラリ追加や複雑化になる

### 手順

1. 実装を止める範囲を限定する。
2. 反対理由を、現行フェーズスコープ、安全性、単純性、可逆性、監査可能性のどれに反するかで説明する。
3. 現行フェーズ内の代替案を1つ提示する。
4. 後続マイルストーンで扱う場合に必要な設計文書、DB/RLS/UI/受け入れ基準を示す。
5. 危険でない周辺作業は止めずに進める。

## 5. エスカレーションworkflow

### 人間確認が必要な条件

AIが依頼者へ確認してよいのは次に限る。

- 現行フェーズスコープを変更する。
- 公開、共有、課金、個人情報、法務、規約、外部サービス契約に触れる。
- Production環境、secret、外部project、billing、plan、URL登録を決める。
- 本番データの削除、破壊、復元不能な変更、不可逆運用を伴う。
- 複数案が同等に安全で、差が純粋にプロダクト方針、好み、優先順位である。

### 質問方法

1. AIが推奨案を1つに絞る。
2. 推奨案が優先した軸を明記する。軸は安全性、スコープ厳守、単純性、可逆性、監査可能性のいずれかにする。
3. 止める作業と止めない作業を分ける。
4. 質問は1論点1問を原則にする。
5. 回答待ちの間、危険でない作業を進める。

## 6. 記録workflow

PRがある場合はPR本文を正式記録にする。PR前またはAI単独作業では、該当タスクの作業記録へ残す。チャットだけを正式証跡にしない。

記録項目は `docs/templates/worklog.md` を基準にする。PRでは `.github/pull_request_template.md` を入口にし、同じ項目をPR本文へ残す。

全変更共通の必須項目:

- 確認対象
- 正本ファイル
- 変更分類と変更対象
- GitHub反映状況
- CI要否判断と結果、未確認理由
- 実行コマンドと結果
- 最低限の整合確認
- 完了判断と未解消事項

大きなコード変更または危険変更で追加必須になる項目:

- 採用方針
- 優先軸
- 根拠文書
- 退けた代替案
- 正本ファイルの証拠抜粋
- 詳細な整合確認の証拠
- 実施した検証
- 実施できなかった検証
- 残リスク
- 局所停止条件
- 人間確認を行った場合の質問と回答

DB / RLS / RPCでは、対象単位、参照したSQL / 手順書 / 証跡ファイル、N/Aにした検証観点と理由、権限境界、RLS / grant / revoke / direct CRUD拒否 / `security definer` / AppError分類の追記事項も残す。

## 7. 完了報告workflow

最終報告では、軽微変更とそれ以外で密度を分ける。

軽微変更では、次を簡潔に示す。

1. 確認対象
2. GitHub反映状況
3. 修正した正本ファイル
4. 最低限の整合確認と実行コマンドの結果
5. CI要否判断
6. 未解消事項

大きなコード変更または危険変更では、次も追加する。

1. 変更ファイル一覧
2. 正本ファイルの証拠抜粋
3. 詳細な整合確認の証拠
4. どの矛盾をどう解消したか
5. worklog更新内容の要約
